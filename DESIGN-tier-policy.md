# Page Tier Policy (V0.8, 2026-05-21)

Every user-facing page (HTML page, RN screen, CLI command, chat surface)
in canopy-mono falls into one of three **tiers**. The tier is declared
in a comment header on the page; future contributors can see at a
glance which pattern they're in.

## TL;DR

| Tier | Rendering | Manifest binding | When |
| ---- | --------- | ---------------- | ---- |
| **T1: Substrate-rendered** | Generated from `renderWeb(manifest)` / `renderMobile(manifest)` | Full — page reads NavModel sections + projects items | List/record + state-gated actions; low custom UX |
| **T2: Manifest-bound** | Hand-coded HTML/JSX | Partial — page reads labels, severities, slash commands, confirm messages from the manifest | Rich UX (custom widgets, multi-section pages) but well-defined data shape |
| **T3: Fully bespoke** | Hand-coded HTML/JSX | None — no manifest declaration | Bespoke widgets / wizards / auth flows with no useful data shape to declare |

**The default is T2, not T3.** A page should be T3 only when none of
its skill calls or surfaces benefit from declaration.

## Why this exists

Pre-substrate, every page in every app hand-coded its labels, its
button text, its confirm prompts, its slash commands. Three apps meant
three different mental models for "how do I add an action?". The
substrate (NavModel V0.1→V0.8) closed that gap for some pages but not
all — and that's *correct*. Some pages have UX that doesn't fit the
substrate's shapes (avatar resize, week-availability grid, OIDC
redirect chains), and forcing them through the substrate would
regress UX.

The trap: "we'll migrate everything eventually." We won't. Some pages
should never be substrate-rendered. The right move is to make the
distinction *explicit* and to keep the **data layer** uniform even
where the **rendering layer** isn't.

## Tier definitions

### T1 — Substrate-rendered

**Definition.** The page's structure is generated by calling
`renderWeb(manifest)` (or `renderMobile(manifest)` for RN) and
iterating the resulting NavModel sections + affordances +
itemActions + sectionActions + fields.

**Examples in the repo today:**
- `apps/tasks-v0/web/dag.html` — dag-tree section, hand-built items
  but every action button comes from `section.itemActions[]`
- `apps/tasks-v0/web/mine.html`, `review.html`, `inbox.html`
- `apps/stoop/web/mine.html`, `privacy.html`
- `apps/tasks-mobile/src/screens/{Workspace,MyWork,Review,Inbox}Screen.jsx`

**When to use:**
- List sections where each row's actions are state-gated by manifest
  `appliesTo` rules
- Record sections with editable fields that map cleanly to Q18
  `view.fields[].patch`
- Pages where chat / web / mobile parity matters (declare once, three
  surfaces follow)

**When NOT to use:**
- Page has custom widgets that need direct DOM/RN control (grids,
  maps, drag-drop)
- Sections have heterogeneous shapes that don't fit a single Q17
  `list` or `record`
- Multi-step user flows (substrate is one-skill-per-affordance)

**Header template:**
```html
<!--
  Tier: 1 — substrate-rendered.
  Manifest section: `<sectionId>` in apps/<app>/manifest.js.
  Rendering:
    - section affordances + itemActions + sectionActions + fields all
      come from the NavModel projection (fetched from /navmodel.json).
    - This page imports no skill IDs directly; the substrate's
      `action.opId` strings are the only skill IDs in flight.
-->
```

### T2 — Manifest-bound

**Definition.** The page is hand-coded but reads from the manifest
for the parts that benefit (labels, severities, slash commands,
confirm messages, localisation keys). The data layer is uniform; the
rendering layer is bespoke.

**Examples in the repo today:**
- `apps/stoop/web/settings.html` — rich custom UX (per-field hint
  text, scope badges), but manifest declares the 4 representative
  fields with `patch` + `labelKey` (V0.4-adopt + Q22)
- `apps/stoop/web/profile.html` — 5 sections of bespoke UX (avatar,
  skills picker, location wizard, mnemonic reveal), but manifest
  declares handle / displayName / holidayMode fields with `patch` +
  `labelKey` + `readSkill` (V0.4 + V0.6 + V0.7-adopt). Sign-out
  button reads `op.surfaces.ui.confirm.message` from the manifest
- `apps/tasks-v0/web/pod-settings.html` — same pattern for
  crew-storage-policy

**When to use (DEFAULT for non-T1 pages):**
- Page has at least one user-facing op the LLM agent might want to
  reference (declare it in the manifest's `surfaces.chat.hint`)
- Page has labels or messages that benefit from localisation key resolution
  (Q22 `labelKey`)
- Page has destructive actions that warrant a substrate-level
  severity hint (Q27 `confirm`)
- Page renders a well-defined data shape (record / list) even if the
  UX around it is bespoke

**Header template:**
```html
<!--
  Tier: 2 — manifest-bound.
  Manifest declares: <ops + fields + verbs>.
  Page renders: <custom layout — list what's bespoke>.
  Page reads from manifest:
    - `<opId>.surfaces.ui.label` / `labelKey` — for localisation
    - `<opId>.surfaces.ui.confirm` — Q27 severity hint
    - <anything else>
-->
```

### T3 — Fully bespoke

**Definition.** No manifest declaration. The page is hand-coded
end-to-end. Acceptable when:

- The page is a bespoke widget (calendar grid, drag-drop board)
- The page is a multi-step wizard whose state machine doesn't fit
  the substrate's "one skill per affordance" model
- The page is an auth flow (OIDC redirect, passphrase prompt, mnemonic
  one-shot reveal) — out of scope for the substrate by design

**Examples in the repo today:**
- `apps/tasks-v0/web/availability.html` — week-grid widget
- `apps/stoop/web/onboard.html`, `restore.html`, `create-group.html` —
  multi-step wizards
- `apps/folio-mobile/src/screens/SignInScreen.js` — OIDC redirect flow

**When to use:**
- The page has no skill calls that benefit from manifest declaration
- OR the skill calls are part of an auth flow / one-shot-read where
  Q27 explicitly defers to app-side

**Justification requirement:** every T3 page must declare *why* in
its header comment. This is the gate — reviewing the justification
is when "should this still be T3?" gets asked.

**Header template:**
```html
<!--
  Tier: 3 — fully bespoke.
  Justification: <why no manifest declaration is appropriate>
                 e.g. "week-grid widget — no substrate shape fits"
                      "OIDC redirect flow — auth concern, out of scope per Q27"
                      "passphrase prompt — security UX, app-side per Tier C audit"
-->
```

## Decision tree for a new page

```
Does the page have a list/record + state-gated actions
+ low custom UX?
    ├── YES → T1
    └── NO  → Does any skill the page calls benefit from
              manifest declaration (chat hint, label, severity,
              slash command)?
                ├── YES → T2  ← default
                └── NO  → T3  (justify why in header)
```

## Migration guidance

### T3 → T2 (the most common upgrade)

When migrating a hand-coded page to manifest-bound:

1. **Declare the ops in the manifest.** Add `surfaces.chat.hint` (required
   by stoop's structural test); add `surfaces.slash.command` if the
   chat agent should accept a slash form; add `surfaces.ui.label` +
   `labelKey` for localisation; add `surfaces.ui.confirm` if destructive.
2. **Update the page** to read labels + confirm from the manifest.
   Use the `createOpBinding` helper (V0.8, see
   `packages/web-adapter/src/createOpBinding.js`) — it gives you
   `confirmAndCall(opId)` that handles the gate automatically.
3. **Update the page's tier header** from T3 to T2 with the new
   manifest bindings listed.
4. **Verify drift canary still passes** — the canary test asserts
   every `callSkill('id')` in the page has a matching manifest op.

### T2 → T1 (rarer, opportunistic)

Only when:
- The page's bespoke layout has accumulated enough manifest
  declarations that rendering from the projection would produce the
  same UX with less code
- OR the page is being rewritten for some other reason

Don't migrate T2 → T1 as a goal in itself. T2 is a stable end-state.

### T1 → T2 (rare downgrade)

If a T1 page accumulates per-page custom UX that fights the
substrate's projection, downgrade to T2 — declare the ops in the
manifest still, but hand-code the rendering. This is fine; the
substrate isn't owed every page.

## Expected end-state

In ~6 months of normal evolution, if this policy holds:

| Tier | Share of pages | Notes |
| ---- | -------------- | ----- |
| T1 — substrate-rendered | 30-40% | List/record + state-gate views |
| T2 — manifest-bound | 50-60% | The new default; rich UX with uniform data layer |
| T3 — fully bespoke | <10% | Bespoke widgets / wizards / auth flows only |

Current state of the repo (2026-05-21): roughly **10% T1 / 15% T2 /
75% T3**. The actionable migration work is **T3 → T2**, not T2 → T1.

## Drift canary — one per app, mechanical enforcement

Every app with a page/screen-side skill-reference surface ships a
drift-canary test. It's the **automated counterpart** to the tier
header comments: humans declare tier intent in the header; the
canary catches forgotten / typo'd / renamed skill IDs in code.

| App | Canary file | Scan target | Pattern | Source of truth for real skill IDs |
| --- | --- | --- | --- | --- |
| **tasks-v0** | `test/page-skill-drift.test.js` | `web/*.html` | `callSkill('<id>')` | 19 skill builders imported, output unioned |
| **stoop** | `test/page-skill-drift.test.js` | `web/*.html` + `web/app.js` | `callSkill('<id>')` | static grep over `src/skills/index.js` + identity-resolver substrate (`packages/identity-resolver/src/*.js`) |
| **tasks-mobile** | `test/screen-skill-drift.test.js` | `src/screens/**/*.jsx` + `src/lib/*.js` | `useSkill('<id>')`, `useSkillResult('<id>', ...)`, `callSkill('<id>')` | tasks-v0's full builder set imported across the workspace |
| **household** | `test/page-skill-drift.test.js` | `web/main.js` + `web/index.html` | `callSkill('<id>')` | `HOUSEHOLD_SKILL_REGISTRY` keys + `agent.register('<id>', ...)` greps in `bin/household-web.js` |

### Apps NOT covered + why

| App | Reason |
| --- | --- |
| **folio** (web) | No HTML pages — folio is HTTP routes + CLI. The folio manifest is declaration-only; pages don't `callSkill`. |
| **folio-mobile** (RN) | RN screens use `engine.foo()` directly (sync-engine-based, not agent-based). The skill-ref pattern doesn't apply. |
| **stoop-mobile** (RN) | Not yet covered. Add when stoop-mobile next gets updated; use the tasks-mobile canary as the template. |

### Canary conventions

Each canary follows this shape:

1. **Strip comments before scanning.** HTML comments (`<!-- ... -->`)
   for web; JS line + block comments for `.js`/`.jsx`. Without this,
   JSDoc examples like `useSkill('id')` inside docstrings show up as
   false positives.
2. **Declare an `ALLOWLIST` set.** Every entry MUST cite (in a comment)
   *why* it's exempted — usually because the skill is provided by a
   substrate package (e.g. `@canopy/identity-resolver` issueInvite /
   redeemInvite) or composed across apps (e.g. tasks-mobile's
   `ProfileMineScreen` calling stoop's `getMyProfile`).
3. **Diff page IDs against real IDs.** The test message lists every
   orphan with a 3-option fix recipe (typo / add skill / allowlist).
4. **Sanity-check non-empty sets.** A separate `it` asserts both the
   page-side and real-side sets have a plausible minimum size, so
   the canary doesn't silently pass if a builder import breaks and
   the real set goes empty.

### What canaries catch

| Bug | Caught |
| --- | --- |
| Typo: `callSkill('clreaInbox')` | ✅ |
| Skill renamed, page missed | ✅ |
| Dead page calls deleted skill | ✅ |
| Cross-app skill leak (e.g. tasks-v0 page calling a stoop skill) | ✅ — flagged unless added to ALLOWLIST with justification |

### What canaries do NOT enforce

| Concern | Where it's enforced instead |
| --- | --- |
| Is the skill in `manifest.operations[]`? | The reverse canary (`sp3-manifest.test.js`-style) — manifest → real skills |
| Is the page's tier (T1/T2/T3) correct? | Human review of the tier header comment |
| Is the skill *used* by some page? | Not enforced (dead skills are a separate cleanup concern) |

### Adding a canary for a new app

1. Copy the closest sibling (`tasks-mobile` for RN, `stoop` for web-only-static-grep, `tasks-v0` for runtime-stub).
2. Wire the skill source path(s) for the app.
3. Run the test. Fix orphans (typo / add skill / ALLOWLIST with reason).
4. Commit alongside any related manifest / page changes.

## Decision rationale

Why three tiers, not two? Because **"hand-coded" and "no manifest
binding" are different concerns**. A T2 page can be 95% hand-coded
HTML + 5% manifest binding (a `labelKey` lookup, a `confirm` gate)
and still benefit from substrate uniformity at the data layer. A T3
page genuinely has nothing to declare. Conflating the two would push
the bar too high (everything-must-be-substrate-rendered) or too low
(any custom JS = T3).

Why is T2 the default? Because the **cost of declaring an op in the
manifest is small** (~5-15 lines) and the **value compounds**:
- LLM agents can dispatch the op without per-page rules
- Drift canary catches forgotten skill declarations
- Cross-page changes (severity colours, localisation keys) flow through
  one source of truth
- Future T2→T1 migration is cheap when it makes sense

T2 is the **sweet spot** where uniformity at the contract layer
coexists with freedom at the rendering layer. That tension —
"uniform but not rigid" — is what this policy makes durable.
