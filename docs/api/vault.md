## `src/OAuthVault.js`

### `OAuthVault`

**Kind:** class · **Import:** `OAuthVault` from `'@onderling/vault'`

```js
class OAuthVault
new OAuthVault({ vault } = {})
```

Typed OAuth-token storage on top of any Vault adapter, keyed `oauth:<service>:<accountId>` with
a `'default'` accountId fallback for single-account use. `getTokens` proactively refreshes when
the access token is within 60s of expiry (via the per-service `RefreshFn` registered with
`registerRefreshFn`); concurrent refreshes for the same (service, accountId) share one
in-flight promise. `refreshTokens` forces a refresh for the reactive 401-fallback path.

**Methods:** `registerRefreshFn()` · `storeTokens()` · `getTokens()` · `refreshTokens()` · `revokeTokens()` · `listAccounts()`

### `makeAuthorizedFetch`

**Kind:** function · **Import:** `makeAuthorizedFetch` from `'@onderling/vault'`

```js
makeAuthorizedFetch(oauthVault, service, accountId, opts = {})
```

Wrap an arbitrary `fetch` so a 401 triggers a single refresh + retry
via OAuthVault, transparently.  This is the "reactive safety net"
half of Q-F.2.

**Parameters**

- `oauthVault` `OAuthVault`
- `service` `string`
- `[accountId]` `string`
- `[opts]` `{ fetch?: typeof fetch }` — optional fetch override (tests).

**Returns:** `(input: RequestInfo|string, init?: RequestInit) => Promise<Response>`

## `src/Vault.js`

### `Vault`

**Kind:** class · **Import:** `Vault` from `'@onderling/vault'`

```js
class Vault
new Vault()
```

Abstract Vault — secure secret storage.

The private key seed is never held in plaintext outside the vault after
AgentIdentity is constructed. All backends implement these five methods.

Key naming conventions (all vaults use these):
  'agent-privkey'             — base64url Ed25519 seed (32 bytes)
  'token:<agentId>:<skill>'   — JSON-serialised capability token
  'group-proof:<groupId>'     — JSON-serialised group proof
  'a2a-token:<url>'           — Bearer token for an A2A peer
  'solid-pod-token'           — Solid OIDC token

**Methods:** `get()` · `set()` · `delete()` · `has()` · `list()`

## `src/VaultIndexedDB.js`

### `VaultIndexedDB`

**Kind:** class · **Import:** `VaultIndexedDB` from `'@onderling/vault'`

```js
class VaultIndexedDB extends Vault
new VaultIndexedDB({ dbName = 'dwag-vault', storeName = 'vault', encryptionKey } = {})
```

Browser `Vault` backed by IndexedDB (string values in a single object store). Optional AES-GCM
encryption at rest: pass `encryptionKey` as a CryptoKey, or as a passphrase string from which a
key is derived. Throws at construction when IndexedDB is unavailable.

**Methods:** `get()` · `set()` · `delete()` · `has()` · `list()`

```js
class VaultLocalStorage extends Vault

```

Browser `Vault` over `localStorage`, namespacing keys under `prefix` (default `'dwag:'`).
Values are stored as plaintext strings; a custom `Storage` (e.g. sessionStorage or a mock) can
be injected. Throws at construction when no storage is available (browser only).

**Methods:** `get()` · `set()` · `delete()` · `has()` · `list()`

## `src/VaultMemory.js`

### `VaultMemory`

**Kind:** class · **Import:** `VaultMemory` from `'@onderling/vault'`

```js
class VaultMemory extends Vault
new VaultMemory()
```

In-memory vault. Secrets are lost when the process exits.
Use for tests and ephemeral server agents.

follow-up B`snapshot()` + `VaultMemory.fromSnapshot()`
support callers that want to persist a vault's contents to a
regular DataSource (e.g. Tasks's cap-token-bound bot agents
serialise their vault under the local-store cache so cap-token
bindings survive a CLI restart). Snapshots are plain JSON-safe
objects (`{key: string-value}`) — encryption-at-rest, if needed,
is the caller's responsibility.

**Methods:** `get()` · `set()` · `delete()` · `has()` · `list()` · `snapshot()`

## `src/VaultNodeFs.js`

### `VaultNodeFs`

**Kind:** class · **Import:** `VaultNodeFs` from `'@onderling/vault'`

```js
class VaultNodeFs extends Vault
new VaultNodeFs(filePath, passphrase)
```

Node.js `Vault` persisting to a single JSON file. With a passphrase, values are encrypted with
AES-256-GCM under a scrypt-derived key; without one the file is plaintext (dev/CI only). Node
built-ins load lazily on first use, so the module itself is safe to load in a browser.

**Methods:** `get()` · `set()` · `delete()` · `has()` · `list()`
