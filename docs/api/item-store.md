## `src/audience.js`

### `audienceFromItem`

**Kind:** function В· **Import:** `audienceFromItem` from `'@onderling/item-store'`

```js
audienceFromItem(item)
```

V0a (2026-05-21) вЂ” `audienceFromItem(item)` bridge helper.

Resolves an item's effective audience by checking, in order:
  1. `item.audience` (the new richer field вЂ” added V0a).
  2. `item.visibility` (the legacy short-hand вЂ” kept forever per
     forward-additive discipline).
  3. Substrate default `'household'`.

Consumers (renderer affordances, audience-aware queries,
`@onderling/circles`' `resolveAudience`) should call this helper
rather than reading either field directly.  That way the legacy
vs. new field choice stays contained.

Item-store stores the field; it does NOT resolve audience to a
member set.  Resolution (string short-hands в†’ webids, circle-ref
walks, union flattening) lives in `@onderling/circles`'s
`resolveAudience(audience, ctx)`.

Forward-additive: a future V0b may add normalisation here
(e.g. canonicalising `'role:admin'` to a structured form), but
the V0a contract is "return the value as-stored".  Don't pre-
normalise вЂ” leave that to `@onderling/circles`.

**Parameters**

- `item` `import('./types.js').Item`

**Returns:** `import('./types.js').Audience`

### `audienceMatches`

**Kind:** function В· **Import:** `audienceMatches` from `'@onderling/item-store'`

```js
audienceMatches(itemAudience, filterAudience)
```

does an item's effective audience satisfy a `ListFilter.audience`
query?

`itemAudience` is the item's effective audience (as produced by
`audienceFromItem`); `filterAudience` is the audience the caller is
querying for ("show me items shared with X").  Returns `true` when the
item should appear in the results.

## Match semantics
The relation is directional вЂ” "does `itemAudience` (the container)
cover the `filterAudience` (the target)?":

 1. **Exact match.**  `itemAudience` structurally deep-equals
    `filterAudience` (key-order-independent).  This is the original
    V0b behaviour and covers every plain short-hand
    (`'circle:A'`, `'household'`, `'private'`, вЂ¦), `set`, `circle-ref`,
    `union`, and `public` value against an identical filter.

 2. **Union membership.**  When `itemAudience` is
    `{kind:'union', of:[вЂ¦]}`, the item matches if the target satisfies
    ANY element of `of` (recursively).  So a query for
    `{kind:'circle-ref', id:'c1'}` вЂ” or the short-hand `'household'` вЂ”
    matches an item shared with `union(household, circle-ref c1)`.

 3. **Set membership.**  When `itemAudience` is
    `{kind:'set', members:[вЂ¦]}` and `filterAudience` is a plain webid
    string, the item matches if `members` includes that webid.  So a
    query for `'webid:bob'` matches an item shared with
    `set(webid:alice, webid:bob)`.

All other cases are `false`.  In particular:
 - `{kind:'public'}` items match ONLY the `{kind:'public'}` filter
   (public is NOT treated as covering every possible target вЂ” that
   would silently broaden every audience query).
 - **Not normalised.**  The string short-hand `'circle:X'` and the
   structured `{kind:'circle-ref', id:'X'}` are still NOT equivalent
   вЂ” normalisation lives in `@onderling/circles`, which item-store can't
   depend on (layering).  A store may inject a `normalizeAudience`
   callback in a later revision to close that gap.

Backward-compatible: a match that succeeded under V0b strict-equality
still succeeds here (rule 1 is unchanged); membership only ever
*adds* matches for `union`/`set` container audiences.

**Parameters**

- `itemAudience` `import('./types.js').Audience` вЂ” The item's effective audience (from `audienceFromItem`).
- `filterAudience` `import('./types.js').Audience` вЂ” The audience being queried for.

**Returns:** `boolean`

### `audienceMatchesAny`

**Kind:** function В· **Import:** `audienceMatchesAny` from `'@onderling/item-store'`

```js
audienceMatchesAny(itemAudience, filterAudiences)
```

cross-circle query predicate. Does an item's effective
audience satisfy ANY audience in a SET (`ListFilter.audiences`)?

A cross-circle query spans MULTIPLE audiences/circles at once вЂ” "show
me items visible to circle A OR circle B OR вЂ¦".  This is the
set-valued sibling of {@link audienceMatches}: the item matches when
`audienceMatches(itemAudience, fa)` holds for at least one `fa` in
`filterAudiences`.  Each element is matched with the full
single-audience contract (exact / union-membership / set-membership),
so the per-audience semantics are unchanged.

## Edges
 - **Empty set** (`filterAudiences === []`) matches NOTHING вЂ” an
   empty union of audiences is visible to no-one, so a query across
   zero circles returns zero items.  (This mirrors `[].some(...) ===
   false` and the `union`-of-nothing intuition.)
 - **Single element** (`[a]`) is exactly `audienceMatches(item, a)` вЂ”
   the single-audience path is a strict special case, kept
   back-compatible.

Item-store keeps `filter.audience` (single) and `filter.audiences`
(set) as independent, composable clauses; nothing about the
single-audience path changes.

**Parameters**

- `itemAudience` `import('./types.js').Audience` вЂ” The item's effective audience (from `audienceFromItem`).
- `filterAudiences` `import('./types.js').Audience[]` вЂ” The audience SET being queried for.

**Returns:** `boolean`

### `causalWinner`

**Kind:** function В· **Import:** `causalWinner` from `'@onderling/item-store'`

```js

```

Decide which side to keep when an inbound item meets the local copy.

**Parameters**

- `local` `object|null|undefined` вЂ” the currently-stored item (has a stamped `updatedAt`), or null/absent
- `incoming` `object` вЂ” the inbound PAYLOAD (its `updatedAt` is the origin clock, if present)

**Returns:** `'incoming'|'local'` вЂ” no local в†’ 'incoming' (first arrival / create) - incoming has no comparable clock в†’ 'incoming' (backward-compat last-received-wins) - local has no comparable clock в†’ 'incoming' (local predates metadata; accept the clock-bearing update) - incoming.updatedAt > local в†’ 'incoming' (causally newer wins) - incoming.updatedAt < local в†’ 'local' (causally OLDER inbound must NOT clobber) - equal updatedAt, higher writer id в†’ 'incoming' (deterministic concurrency tiebreak) - otherwise (fully equal / lower) в†’ 'local' (idempotent: no rewrite, no churn)

## `src/circleStoreInbound.js`

### `wireCircleStoreInbound`

**Kind:** function В· **Import:** `wireCircleStoreInbound` from `'@onderling/item-store'`

```js
wireCircleStoreInbound({ notifyEnvelope, store, kind = 'household-item', removed = 'household-item-removed', prefix, } = {})
```

wireCircleStoreInbound вЂ” ingest peer sync envelopes into a CircleItemStore (cluster L3 В· no-pod-sync INBOUND).

The counterpart to publish-on-write (`setSyncHook` / `wireStoreMirror`): subscribes to the notify-envelope
substrate and applies inbound items to the per-circle store. Because the CircleItemStore PRESERVES the remote
item id (`put` keeps `item.id`), sync is trivially idempotent вЂ” the same id on every device, so a received
item is just `put` (create-or-replace) and a removal is `delete` by id; no `syncedFromId` bookkeeping (the
legacy ItemStore needed it because it minted its own ids). Writes use `{sync:false}` so an ingest never
re-publishes the same item back to the mesh (the echo loop). Filtered by the envelope `ref` prefix в†’ per-circle.

Ingest is CAUSAL, not last-received-wins (Objective L): writes go through `put(payload, {sync:false, origin:true})`,
which PRESERVES the payload's origin `updatedAt`/`updatedBy` (instead of re-stamping to local ingest time) and
keeps the causally-newer side вЂ” a peer's OLDER edit can no longer clobber a newer local one just because it
arrived later; true concurrency resolves by a deterministic writer-id tiebreak. This is origin-timestamp +
writer-id last-writer-wins at item granularity (not a field-level merge); payloads without origin metadata fall
back to last-received-wins so pre-metadata peers still ingest. See `causalMerge.js` for guarantees/limits.

**Parameters**

- `args` `object`
- `args.notifyEnvelope` `{subscribe:(opts:{kind:string,callback:Function})=>(()=>void)}` вЂ” @onderling/notify-envelope (injected)
- `args.store` `{put:Function, delete:Function}` вЂ” the CircleItemStore for this circle
- `[args.kind='household-item']` `string` вЂ” add/update envelope kind
- `[args.removed='household-item-removed']` `string` вЂ” removal envelope kind
- `[args.prefix]` `string` вЂ” only ingest envelopes whose `ref` includes this (per-circle scope)

**Returns:** `() => void` вЂ” unsubscribe

### `makeCanonicalShareHook`

**Kind:** function В· **Import:** `makeCanonicalShareHook` from `'@onderling/item-store'`

```js

```

CANONICAL share hook (objective L) вЂ” the WRITE-side companion for the `canonical` posture. Where
`makeShareGrantHook` grants a plain ACP read and (optionally) re-seals a COPY, this instead composes the
injected `createCanonicalShare` controller (`@onderling/pod-client`) so a share GRANTS the recipient INTO the
item's group-key resource (`grantMember` в†’ O(1) key re-wrap) AND lands the ACP read grant вЂ” NO copy is
written; `shareIntoAudience` still writes ONLY the `shared-ref` pointer. The recipient then opens the

exactly as `sharing`/`seal` are injected for the copy postures. item-store only orchestrates the per-

**Parameters**

- `opts` `object`
- `opts.canonicalShare` `{ share:Function, revoke:Function }` вЂ” a `createCanonicalShare(...)` controller.
- `[opts.currentRecipients]` `string[]|(()=>string[]|Promise<string[]>)` вЂ” the EXISTING roster's sealing PUBLIC KEYS (the origin circle's members already holding the group key). A thunk is resolved at share time (the live roster). Default: none.

**Returns:** `{ onShare: Function, revoke: Function }`
