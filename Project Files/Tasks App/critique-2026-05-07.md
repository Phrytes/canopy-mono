# Tasks V1 — design critique (2026-05-07)

> Companion to [`advice-2026-05-07.md`](./advice-2026-05-07.md).
> Honest pushback on the design before we commit to a coding
> plan. Goal: surface the parts I'm least confident about so we
> either fix them now or accept the trade-offs explicitly. Order
> is rough priority — "most likely to bite" first.

## TL;DR

The advice doc is mostly sound, but **V1 as scoped is probably
too big for the team's velocity, and over-architected for the
simplest real user (a household)**. My biggest worry: we ship a
generalized framework that fits nobody well because we tried to
fit decentralized OSS projects + neighborhood maintenance +
households all in one go. The most useful single change to the
plan: **split V1 into V1.0 (household-shaped, self-mark only,
single crew, no calendar) and V1.1 (decentralized-project
shape — adds approval, multi-crew, calendar, sub-tasks)**, and
let V1.0 ship and get used before deciding V1.1 details.

The other concrete picks in this doc are smaller and cheaper:
sub-task depth cap, revoke softening, calendar via subscription
URLs instead of OAuth-listener bridges, custom roles deferred,
test coverage budgeted explicitly. None invalidate the design;
all should be folded into the implementation plan rather than
discovered during it.

## 1. V1 is probably too big

**The size estimate.** V1 as scoped:

- Crew envelope (config loader, group binding, per-crew skill
  vocabulary, switcher UI).
- DoD lifecycle (`submitted` + `rejected` states, three new
  fields, four new skills, role-policy table extension).
- Sub-tasks (`parentTaskId`, `addSubtask`, DAG-tree helpers, UI
  affordances).
- Master + revoke (new field, new skill, cascade semantics).
- Calendar (read adapter + Folio sync agent + privacy filter).
- In-app inbox (notifier channel + cold-boot via CachingDataSource).
- Custom-role activation (Q-H4.7 (c) extension path).
- 7 workspace screens (workspace, add-task, my-work, review-queue,
  DAG, crew+settings, inbox).
- Localisation (nl + en).
- Closed-beta privacy notice, archive flow, pause flow.

**Realistic estimate**: 4-6 weeks of focused work. Parallels
Stoop V1's 30-35 dev-days post-honesty-pass.

**What I'm worried about.** Stoop V1 originally ended at Phase
10 ("closed-beta hardening") and the honesty pass found the
user-to-user loop wasn't actually shippable; Phases 11-15 added
~10-15 more days. Tasks V1 has a comparable user-loop
requirement (approver + assignee back-and-forth, sub-task
spawning, calendar opt-in) and I'm probably making the same
error.

**Recommendation: split V1 into V1.0 + V1.1.**

- **V1.0** (1.5-2 weeks, household-shaped):
  - Single crew (implicit; no invitation flow).
  - DoD field; `approval: 'self-mark'` only.
  - 3 screens: workspace, my-work, settings.
  - In-app inbox for `missed-deadline` and `task-completed`.
  - No calendar. No sub-tasks. No custom roles.
- **V1.1** (3-4 weeks, decentralized-project shape, after V1.0
  has 4 weeks of real use):
  - Multi-crew envelope + invitation flow.
  - `creator` and `webid:` approval modes.
  - Calendar (subscription-URL-only first; see § 4).
  - Sub-tasks at depth 2.
  - Review-queue + DAG screens.
  - Custom-role UI (deferred from V1.0 entirely).

## 2. Crew is over-architected for households

A household has 4 people who all sleep under the same roof.
Stoop's group-with-invitation-and-GroupProof model is designed
for **closed groups across organisational boundaries** — that's
overkill for "Anne, the author, the kid, and the printer share a
chore list".

V1 forces a "create crew → invite members via QR → redeem
invite → exchange GroupProofs → finally add a task" wizard
before the household can do anything useful. That's a 5-minute
friction wall before first value, in a use case where Asana /
Trello / a paper list win on time-to-first-value in 30 seconds.

The architecture that's right for decentralized OSS projects is
wrong for households. We're letting the harder use case dictate
defaults.

**Recommendation.** V1.0 ships a "household mode" — single
implicit crew per app instance, members added by webid (or even
just display name + ad-hoc keypair if no pod is configured), no
invitation flow at all. The full closed-group + GroupProof
model is a V1.1 toggle at crew creation time ("invitation-only
crew" vs "open crew"). The plumbing exists; it just doesn't
have to fire on every install.

## 3. Generalized-first is an inversion of the rule of two

You said Tasks "is more like a generalized version on top of
which we will create more specialized but compatible apps
later." This is the inversion of the substrate-first rule — we
generalize *after* two concrete consumers want the same thing.
Generalizing first risks committing to compromises (5 standard
roles + custom extension; per-crew everything; DoD optional but
load-bearing) that fit nobody specifically.

Stoop is already a "specialized app" of the underlying
primitives (`item-store` + `skill-match` + closed-group
governance). Household is another. They specialise differently
because the use cases differ. Tasks-as-generalization risks
being everyone's second-best choice because no real first user
shaped its concrete details.

**Recommendation.** Build Tasks V1.0 against ONE concrete
consumer — pick household chores OR a decentralized-OSS-project
that's actually starting (the NLnet PoC itself? a buurttuin
maintenance crew somebody runs?). Generalize when the second
real consumer says "I want what they have but with X different".
Don't ship generalization-on-spec.

## 4. Calendar sync via Folio is a substantial sub-project

Google Calendar push-notification listener:

- Webhook receiver (HTTP endpoint, public-reachable).
- OAuth flow (consent screen, scope handling, refresh-token
  rotation).
- Subscription-watch lifecycle (create, renew before expiry,
  delete on disconnect).
- Change-event parsing (Google's "incremental sync" model with
  syncTokens).
- ETag-based diff resolution against the pod-side `.ics` file.

This is 1-2 weeks alone, before Outlook (similar shape) and
iCloud + CalDAV (polling). To ship V1 with the freebusy user
story working, all four sources need to work *or* the freebusy
story has to degrade.

**Recommendation: V1.0 ships read-only iCal subscription URLs
only** (`webcal://calendar.example.com/feed.ics` as a
copy-paste field). Most calendar tools have a "share calendar
as ICS URL" option:

- Google Calendar: Settings → "Public address in iCal format" or
  the per-calendar secret URL.
- iCloud: "Make Calendar Public" gives a webcal:// link.
- Outlook: "Publish a calendar" gives an .ics URL.
- Generic CalDAV: most servers expose a `/cal.ics` endpoint.

Trade-off: not real-time, no sub-second freshness; refreshes via
poll (15 min default, configurable). For "is X free Friday
afternoon?" this is fine. The OAuth listener bridges become
V1.5 work, scoped against actual usage data ("how often do
users want sub-15-minute freshness?").

This is a 1-day implementation in Tasks itself; no Folio
dependency. Folio still wants the full bridge eventually for the
import-bridge story, but V1 doesn't block on it.

## 5. DoD + approval friction in households

Households don't need approval for "take out the trash". OSS
projects do need approval for "merge this PR". The same default
can't serve both.

If V1 defaults `approval: 'creator'` for any task → households
feel the friction immediately, half the household tasks will
have approval cycles nobody wants, V1.5 changes the default and
everybody retrains.

If V1 defaults `approval: 'self-mark'` → OSS-project users have
to opt every-task into approval, which is fine because they
*want* the formality.

**Recommendation.** V1 hardcodes `approval: 'self-mark'` as the
default. `creator` and `webid:` modes are explicit per-task
choices in the composer (a small dropdown). The user community
that wants approval-by-default can have a per-crew default
(*"this crew defaults to approval-required"*) in V1.1; V1.0
ships only the per-task override.

## 6. Sub-tasks: depth and UI legibility

The advice doc says depth-6 soft warning. That's a guess; I
have zero data.

DAG view legibility past depth 3 is empirically bad —
Trello/Jira don't go deeper without surrendering to swim-lanes
or boards. The terminal-style indented-list mockup I drew
breaks down at depth 4.

**Recommendation.** V1 hard-caps sub-task depth at **2**
(parent + one level of children). V2 unlocks depth 3+ once we
have observed how the depth-2 case is actually used. We are not
starting with users who have asked for depth 6; we're guessing
they might want it.

**Edge case to design for V1.0 still**: what happens when a
sub-task creator leaves the crew? The advice doc punts ("master
becomes parent's master, ultimate fallback admin"). Test that
end-to-end before V1 ships, or it'll fail in production with
data loss.

## 7. Revoke is socially sharp; soften the UX

`revoke` (master yanks the assignee) is a real power asymmetry.
In a business it's normal. In a household ("mom revokes dad's
chore"), in a volunteer crew, or in a decentralized OSS
project ("maintainer yanks contributor"), it lands harder.

Cheap fix:

- `revoke` requires a reason (mandatory free-text field).
- The notification to the assignee includes the reason.
- The assignee can `appeal` (just opens a chat thread back to
  the master via `chat-agent`) — no decision logic, just a
  channel for the conversation.

Adds maybe 50 lines of code. Changes the social tone of the
feature considerably.

## 8. The in-app inbox needs a cold-boot story

I said the inbox lives at `<user-pod>/inbox/`. If the user
opens the app **before** signing into a pod (typical first
launch), the inbox is empty.

Stoop already solved this with `CachingDataSource` +
`SyncCadence` (substrate-candidates, flagged for promotion).
Without them: cold-boot Tasks shows an empty inbox until the
pod connects, which is bad UX *and* hides delivery failures
(the user won't notice that no notifications are arriving
because they're not being recorded anywhere they can see).

**Recommendation.** V1 mandates `CachingDataSource` for the
inbox from day 1 — pull it from Stoop into substrate now (rule-
of-two satisfied: Stoop + Tasks). Don't pretend the offline-
first dance is somebody else's problem.

## 9. Custom roles via pod-edit will hurt

Q-H4-roles-secondary lock said custom roles are configured by
editing `<crew-pod>/crews/<id>/config.json` by hand. For OSS-
project admins (technical users) this is OK; for neighbourhood-
maintenance admins (mixed-trust, possibly non-technical) it
isn't.

**Recommendation.** V1.0 ships the standard 5 roles only — no
custom-role activation at all. Q-H4.7 (c) extension lands in
V1.1 with a small UI ("Add custom role: name, can-claim,
can-approve, can-reject"). The forcing function is the second
use case (neighbourhood maintenance) actually needing it.

## 10. Calendar privacy isn't truly private

"Busy intervals only, no titles" leaks more than it sounds:

- Recurring 1-hour block on Tuesdays at 14:00 → identifiable as
  "doctor's appointment" or "therapy".
- A 2-week solid-busy stretch in summer → identifiable as
  "vacation" (and roughly when).
- Pattern-matching across multiple weeks → identifiable as
  "kids' school schedule" (which schools, which grade).

For closed-beta this is acceptable. For public release it's a
privacy footgun that will turn into a complaint.

**Recommendation.** The closed-beta privacy notice carries an
explicit paragraph spelling this out: *"your busy intervals
reveal your patterns, even without titles"*. V2 supports
**hidden blocks** — events the user marks as `X-PRIVACY: hidden`
in their calendar app are stripped from `getFreeBusy` output
entirely (those slots appear as "free" instead of "busy",
trading honesty for privacy). Not V1; flag it.

## 11. The "everyone has a Solid pod" UX

Stoop already wrestled with this — default issuer + Inrupt
auto-provisioning. Tasks inherits the solution, but also
inherits the unsolved part: cap-token UX feels error-prone
(per project memory:
[`project_capability_tokens_to_inrupt.md`](../../../.claude/projects/-home-frits-expotest-nkn-test/memory/project_capability_tokens_to_inrupt.md)).

V1's three new cross-pod read flows (calendar, skills,
deliverables) re-introduce the same UX surface in three more
places.

**Recommendation.** Hold the V1 implementation plan until the
Inrupt-migration timeline is clearer. If the Inrupt switch is
imminent, ship Tasks V1 *after* it lands so we're not building
three more cap-token UXes that get rewritten next quarter. If
the Inrupt switch is far off, accept that V1 will inherit the
same pain — and bake "fix this in the Inrupt cutover" into the
V2 plan upfront.

## 12. Test coverage isn't budgeted

V1 doubles the surface area; V0's 24 tests are nowhere near
enough.

**Concrete test budget** (specifying so the implementation plan
includes them up-front, not bolted on later):

| Area | Estimated tests |
|---|---|
| Lifecycle state-transition matrix (5 states × ~6 transitions × 2 approval modes) | ~15 |
| Role-policy gate per role × action (5 roles × ~10 skills) | ~25 |
| DAG sub-task cycle detection (parent → child → grandchild → parent) | ~5 |
| Calendar privacy filter (visible vs masked vs hidden) | ~5 |
| In-app inbox cancellation cascade (item-completed → cancel-pending-nudges) | ~5 |
| Revoke cascade (revoke parent → orphaned children → reassign / cancel prompt) | ~5 |
| Crew-config loading (skill vocab, role table, custom roles, missing) | ~5 |
| Calendar subscription URL fetch + parse | ~5 |
| **Total before UI smoke tests** | **~70** |

UI smoke tests (web): ~15 more. Total V1: ~85 tests. Plan
budgets ~3-4 days for tests; otherwise they ship light and we
regress.

## 13. Observability and feedback loops are missing

Stoop called out "feedback loop on UX-load-bearing UX" as
engineering practice. Tasks should match.

UX-load-bearing tunables in Tasks:

- Deadline-reminder cadence (24h before? 1h before? both?)
- Missed-deadline notice timing (at deadline? 1h after? next morning?)
- Approval-prompt cadence (immediate? batched daily? both?)
- Sub-task depth warning threshold (we're starting at "no warning, just refuse beyond 2", which is a punt)
- Calendar-share opt-in friction (default off feels right but might leave too much unset)
- Revoke notification tone (immediate vs digest?)

**Recommendation.** V1 ships per-crew metrics (locally
aggregated; opt-in to share with admin):

- Tasks added / claimed / completed / revoked / rejected /
  approved / missed-deadline.
- Time-to-claim (median, p90).
- Time-from-submit-to-approval (median, p90).
- Notification arrival vs. user-action latency.

Surface these in the crew settings → "stats" tab. Adjust
defaults at V1.5 / V2 based on what real crews do. Don't pretend
to know the right cadences from a spec.

## 14. The chat-bot forward-compat note hides actual work

"Don't lock anything that prevents future bot access" sounds
free. It isn't:

- Bot agents need cap-token-bound identity (Stoop V2 substrate-
  candidate, not yet shipped).
- Bot skills need separate dispatch from UI skills (so
  *"approve task"* via Telegram doesn't trigger a UI redirect).
- Bot users need a different identity in the audit log (so
  "Anna via Telegram" reads differently from "Anna via web").

**Recommendation.** Either (a) commit chat-bot to V1.5 with
budgeted scope (~1 week), or (b) remove the forward-compat note
and accept that future bot work might need V1 schema changes.
The current "we're forward-compat" framing is a hidden
commitment we'll regret.

## 15. We're rebuilding what some commercial SaaS already does

Tasks-with-skills + DoD + approval + DAG ≈ Asana / ClickUp /
Linear. We're NOT trying to compete on feature surface; we ARE
trying to be decentralized + pod-backed + self-hostable.

Risk: V1 users compare to Asana, find Tasks lacking, churn.
The decentralization differentiation only matters for users who
already care.

**Recommendation.** Pick the early-adopter user explicitly.
"For technical users who want to self-host their team's task
ledger and own the data" sets expectations. "Asana but
decentralized" doesn't — too aspirational, invites bad
comparisons. The README + privacy notice should both lean into
the niche framing.

## What's right about the design

So this isn't a one-sided pile-on. The advice doc gets several
things right:

- **DoD + approval lifecycle on `item-store`** is correctly
  scoped — additive, doesn't break V0, satisfies rule-of-two
  with Stoop's lend-flow as the second consumer. Schema additions
  are minimal; the new states fit cleanly.
- **Per-field split within the crew pod** is the right answer
  to the README's open question 1. The Phase 5.2-7 substrate
  refactor makes this trivially achievable.
- **In-app-inbox-first** is the right notification posture.
  Calmness > chrome; doesn't depend on push provisioning.
- **Skill-import from user pod** with prefilled-edit-before-
  submit is a pattern worth standardizing across the suite.
  Cheap, big UX dividend, naturally cross-app.
- **Pod-data-sharing caution principles** as a process rule
  (rather than a code rule) is the right discipline. Cross-pod
  flows will keep accruing; having a sign-off gate prevents
  drift.
- **Mapping V0 substrates to V1 without inventing new packages**
  is the right call. The substrate refactor work is paying off
  here — extensions to `item-store` + a new channel on
  `notifier` is structurally minimal.
- **The Stoop-advice structural mirror** (TL;DR → what exists →
  per-input deep dive → keep/adapt/drop/defer → substrate fit →
  user stories → phased plan → open questions) is paying off.
  Easy to scan, easy to compare against Stoop.

## What I'd do differently — concrete recommendation

If I had a free hand on the implementation plan:

1. **V1.0 (1.5-2 weeks): household chore tracker.** Single
   implicit crew, no invitations, `approval: 'self-mark'` only,
   3 screens (workspace + my-work + settings), in-app inbox,
   cached data source for offline boot. Ship to one real
   household; observe.
2. **V1.1 (3-4 weeks): decentralized-project shape.** Add
   multi-crew envelope, invitation flow, `creator` /
   `webid:` approval modes, calendar via subscription URLs,
   sub-tasks at depth 2, review-queue + DAG screens, master /
   revoke (with reason + appeal). Drive by one real OSS-style
   crew (the NLnet PoC itself? a small contributor team?).
3. **V1.5 (later, demand-driven): the deferred items.** Custom-
   role UI, sub-task depth >2, OAuth-listener calendar bridges
   in Folio, push-when-relay-ready, chat-bot bridge,
   compensated-role flag.

Each step is forced by an actual second consumer. We don't
generalize on spec.

The advice doc's design isn't wrong. It's just bigger than what
we can validate before committing. Splitting it into V1.0 +
V1.1 keeps the same end-state but gets us a real user 4 weeks
earlier — and that user's reaction will tell us whether V1.1's
shape is right before we build it.
