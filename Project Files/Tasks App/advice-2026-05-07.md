# Tasks (Klus) — integration advice + high-level design (2026-05-07)

## Functional summary

**What is the Tasks app?** A shared to-do list for a group of
people (and their devices), where every task can carry skill
requirements, a deadline, and a definition of done. When one task
finishes, anything that depended on it becomes available to the
next person who has the right skills. People work together
across organisations: a household, a friend group, a business
team, a neighbourhood-maintenance org, or a decentralised OSS
project — all use the same app, with role-aware permissions
tuned to the group. Each person's data lives in their own Solid
pod; the group's task ledger lives in a pod owned by whoever
makes sense for that group.

**What makes it different from Stoop?** Stoop ends a thread when
two people connect. Tasks adds a **completion moment** (mark
done, optionally approved by the issuer) — and that completion
event is what unlocks dependent tasks for the next person. Tasks
is for *coordinated work that ends in a result*; Stoop is for
*finding the right neighbour for a one-off favour*.

## TL;DR

- **App name stays "Tasks".** Tasks is the generalised app on top of which more specialised apps (Stoop already, Household already, future ones) are built. The *envelope concept* inside Tasks is a "Crew" (working term) — what Stoop calls a Group, Tasks calls a Crew because a household isn't a project and isn't quite a group either.
- **Tasks V1 ≈ `apps/tasks-v0` + Crew envelope + DoD-with-approver lifecycle + sub-task spawning + local-only calendar conflict view + in-app inbox + issuer notifications.** Most moving parts already ship; one new substrate (`@canopy/local-store`, lifted from Stoop) plus extensions to `item-store` and `notifier`.
- **Local-only operation is a hard rule.** The whole app must boot, run, add tasks, claim, complete, and notify *without a pod connection*. Pod sync attaches when configured, never blocks the boot path. Same calmness Stoop achieves via `CachingDataSource`.
- **Calendar matching is local only.** No network freebusy skill, no cross-pod calendar reads. The user reads their own pod-mirrored (or local) `*.ics`; the claim flow shows them their conflicts on the proposed deadline. Other crew members never see your busy intervals.
- **Calendar-from-external-source moves to `apps/import-bridge-v0`.** Folio gets a simple "read .ics from the pod" capability for display purposes; the OAuth + listener + change-event work lives in import-bridge as one more source connector. Tasks V1 ships with mockup `.ics` fixtures so testing + dev-mode work without depending on the import-bridge work.
- **DoD + approver is a lifecycle extension** on `item-store`, not a new substrate. Two new states (`submitted`, `rejected`) + four optional fields (`definitionOfDone`, `approval`, `deliverable`, `master`, `parentTaskId`). Default `approval: 'self-mark'` everywhere; `creator` and `webid:` are explicit per-task picks.
- **Issuer notifications go through the in-app inbox.** Push is wired but feature-flagged off until V1.5 (no Firebase yet). In-app inbox is calm by default and works without any push infrastructure.
- **Sub-tasks by the accepter, no hard depth cap.** Claimer can spawn dep-subtasks; each crew config sets `subtasks.adminApprovalDepth` (default 3) — spawning past that depth queues an admin-approval request rather than failing.
- **One master per task** — the master (issuer or admin/coord) can `revoke` an assignment with a **mandatory reason**; the previous assignee can `appeal` (opens a chat thread back to the master). Distinct from `release` (assignee triggers) and `reassign` (master moves directly).
- **Skill-import from canonical user-pod profile** with prefilled-edit-before-submit form — same pattern across Tasks / Stoop / Household / Folio. Cross-app substrate candidate flagged.
- **Custom roles deferred to V1.1.** V1 ships standard 5 roles only.
- **Chat-bot bridge committed to V1.5** with budgeted scope (~5 days). Not "forward-compat hand-waving" — explicit deferred phase.
- **Observability ships in V1.** Per-crew metrics (admin defaults, user overrides) so cadence tunables can be tuned from real data.
- **One new SDK substrate is promoted** in V1: `@canopy/local-store` (lifted from Stoop's `CachingDataSource` + `SyncCadence`; rule of two satisfied by Stoop V2 + Tasks V1). Plus extensions to `item-store` (DoD lifecycle + sub-tasks) and `notifier` (`InAppInboxChannel`).

## The V1 inputs, in depth

### Input 1 — Calendar / availability (revised 2 — local matching only)

**The need.** When a task has a `dueAt`, the claimer benefits
from *"is my Friday actually free?"* — a hint while deciding
whether to take it on.

**The honest take, refined again (locked 2026-05-07).** Two
changes from the previous draft:

1. **Calendar matching is purely local.** No `getFreeBusy`
   network skill. The user's calendar stays on their own pod (or
   a local file when no pod is connected); only *they* see their
   conflicts. Other crew members never see your busy intervals,
   and no requesting / freebusy-aggregation goes over the wire.
   This is a meaningful feature loss (the assigner can't pre-
   filter candidates by who's free) in exchange for a meaningful
   privacy win (calendar data never leaves the user's device).
2. **Calendar-to-pod sync belongs in the import-bridge project,
   not Folio.** Folio's job is to read `*.ics` from the pod when
   needed — period. The OAuth + listener + change-event
   mechanics for Google / Outlook / iCloud are exactly the
   shape `apps/import-bridge-v0` already exists for; one more
   source added there.

**Architecture in three layers.**

```
   ┌─────────────────────────────────────────────────────────┐
   │ External calendar  (Google / Outlook / iCloud / CalDAV) │
   └─────────────────────────────────────────────────────────┘
                             │  (OAuth + listener-or-poll)
                             ▼
   ┌─────────────────────────────────────────────────────────┐
   │ apps/import-bridge-v0  — calendar source connector      │
   │ writes <user-pod>/calendar/<source>.ics                 │
   └─────────────────────────────────────────────────────────┘
                             │
                             ▼
   ┌─────────────────────────────────────────────────────────┐
   │ <user-pod>/calendar/*.ics  (or local file in local mode) │
   └─────────────────────────────────────────────────────────┘
                             │  (read-only)
                             ▼
   ┌─────────────────────────────────────────────────────────┐
   │ apps/tasks-v0  — local iCal reader, conflict-only view   │
   └─────────────────────────────────────────────────────────┘
                             │
                             ▼
                       Claimer's UI
                       (only the claimer sees their conflicts)
```

**The sync side moves out of this app entirely.** Tracked in
`Project Files/projects/03-import-bridge/README.md`. Tasks V1
does **not** depend on the sync side being built — without it,
the user can still manually drop `.ics` files into their pod (or
use a Solid-calendar tool like Penny that already writes the
same shape), and Tasks reads them.

**The read side, in Tasks V1.**

```ts
// app-internal helper, NOT a registered skill
import { readMyCalendar } from './calendar/iCalReader.js';

const conflicts = await readMyCalendar({
  // pod-mirrored ics files OR a local file in local-only mode
  source: { kind: 'pod', client: podClient, container: '<user-pod>/calendar/' },
  range:  { start: dueAt - 24*3600*1000, end: dueAt + 24*3600*1000 },
});
// conflicts: [{ start, end, summary?, source }, ...]
// Returned only to the local UI; never sent to other agents.
```

- Pure local read; no network skill registration.
- Parses iCal with `ical.js` (RRULE expansion, timezones,
  VTIMEZONE).
- Returns conflicts to the local claim-flow UI for *this user
  only*.
- **Local-mode fallback** — if no pod is connected, reads from
  a local `~/.tasks/calendar/*.ics` directory. Same parser,
  same shape; no other code-path differences. (See *Local-only
  mode* below.)

**Mockup calendar fixture for testing.** V1 ships a tiny
`*.ics` fixture under `apps/tasks-v0/test/fixtures/calendar/`
(a couple of recurring events, a one-shot, an all-day) so that
(a) tests have predictable input and (b) developers can
exercise the conflict-view path without configuring a real
calendar source. The pod-mock loader treats this directory as
the pod's `/calendar/` for dev-mode launches.

**What the Tasks app does with it (V1):**

- When the user taps "claim" on a task with a `dueAt`, the UI
  shows *"You have N events around this time: [list]"* — purely
  informational. The user decides.
- The task composer's assignee picker does **not** pre-filter by
  candidate availability. (The previous draft had this; it's
  removed because it requires fetching other members' freebusy.)
- Out of scope for V1: auto-scheduling, write-side calendar
  events, cross-member availability hints.

**Where it lives.**

- **Read-side iCal reader** — `apps/tasks-v0/src/calendar/iCalReader.js`,
  app-internal. Pure function; no substrate flag.
- **Sync side (calendar-to-pod connector)** — moved to
  `apps/import-bridge-v0` as one of its source connectors.
  Tracked under project #3.

**Compatibility with the SDK + substrates.** Zero new SDK
primitives required. Pure local helper. Solid-calendar tools
(SolidOS / Penny / Cozy) that write `*.ics` to the pod
continue to work — they just put files where Tasks reads from.

### Input 2 — Skills + deadlines per crew

**Already partly there.** Tasks-v0 items already carry
`requiredSkills: string[]` and `dueAt: number`. What's *not*
there is the per-crew layer, and the cross-app skill-profile
seam:

#### Importing the user's existing skills from their pod

A user who has already declared their skills in another app
(typically Stoop, possibly the household app) shouldn't have to
type them again. The right pattern:

- Each user has a **canonical skills profile** at
  `<user-pod>/profile/skills.json`. Schema:
  `[{id, label, description?, level?, lastUsed?}]`. *Level is
  free-form ("beginner" / "advanced" / numeric); apps can
  interpret as they like.* This file is owned by the user and
  not by any one app.
- When an app needs the user's skill list (Tasks "what skills do
  you bring to this crew?", Stoop "list your skills in the
  Oosterpoort group", Household "what can you help with?"), it
  **reads the canonical profile and shows a prefilled,
  editable form**. The user reviews, ticks/unticks, edits or
  adds, and submits.
- The submit writes two places:
  1. The **app/crew-specific list** (`<crew-pod>/crews/<id>/skills/<webid>.json`
     for the crew-scoped projection that Tasks uses).
  2. **Optionally back to the canonical profile**, gated by a
     "save these new skills to my profile so other apps can use
     them too?" checkbox (default OFF for added skills, ON for
     edits to existing skills — least-surprise direction).
- Posture per crew remains in
  `<member-pod>/posture/<crewId>.json` (per-crew, not
  canonical) — your willingness changes per context, your
  capabilities don't.

**Why this pattern.**

- One source of truth for "things I can do"; per-app
  customisation lives on top, not instead.
- Adding a fifth crew doesn't mean re-typing your skills.
- The user retains control — every cross-app read goes through
  the prefilled-form-edit-before-submit step, never silent
  mirroring. *This is a load-bearing instance of the pod-data-
  sharing caution principles below — ask once per app, not
  once per session.*
- Pure additive: apps without the canonical profile (or users
  who decline) fall back to typing skills directly, as today.

**Substrate location.** The canonical-profile read + prefilled-
form-shape helper is a candidate for `@canopy/identity-resolver`
(it already manages the user-skills + display-name surface for
its own purposes; lifting the prefilled-form helper there is
additive). **Rule of two**: Tasks V1 + Stoop V2 + Household = 3
consumers — promote in V1.5. App-level for V1 if it lifts later;
substrate-extension if the author wants it lifted from day one.

**Cross-app implications** — this pattern is **also relevant
for Stoop, the household app, and any future skill-shaped app**.
Pointers added to:

- `Project Files/projects/02-neighborhood-app/README.md` (Stoop)
- `Project Files/projects/04-tasks-app/README.md` (this app)
- `Project Files/projects/07-household-app/README.md` (Household)
- `apps/folio/README.md` (Folio is the natural app for editing
  the canonical profile)

#### Per-crew skill vocabulary (the original input 2)

- **Crew skill vocabulary** — the *crew*-side counterpart to the
  user's canonical profile. Each crew carries
  `<crew-pod>/crews/<id>/skills.json` listing the skills the crew
  cares about (e.g. `"frontend"`, `"design-review"`,
  `"on-call"`). The composer auto-suggests from this list;
  free-form skills are still allowed but flagged as
  off-vocabulary. The skill-import prefilled-form intersects the
  user's canonical profile with the crew vocabulary so the user
  sees their relevant skills first.
- **Deadline conventions.** Each crew carries default reminder
  cadences (`"remind 24h before"`, `"missed-deadline notice"`,
  `"escalate at T+0"`) — the notifier wiring reads these instead
  of hard-coding 24h.
- **Skill posture per crew.** A member can mark `frontend:
  always` for one crew but `frontend: negotiable` for another
  (different commitments). Stored in
  `<member-pod>/posture/<crewId>.json`.

**Pod schema additions** (small):

```
<crew-pod>/crews/<id>/
  config.json         # name, members, role table, custom roles
  skills.json         # vocabulary: [{id, label, description?}]
  cadences.json       # deadline-reminder + missed-deadline + completed defaults
  tasks/...           # the existing item-store layout
```

**Why per-crew, not per-app.** A person in three crews has three
distinct skill profiles — the OSS crew doesn't care that they
can plaster a wall. Per-app vocabulary collapses this; per-crew
preserves it. Same logic Stoop applies to per-group reveal state.

**Compatibility.** Pure additive; existing single-household
tasks-v0 keeps working with no crew ID (defaults to the implicit
"household" crew). All extensions sit in `item-store`'s optional
fields + new app-level config files.

### Input 3 — Aimed at decentralised work groups (Crews)

**The shaping requirement.** A "decentralised crew" in this
context spans organisational boundaries: paid contractors,
volunteers, member-companies, contributing individuals,
automated agents (build bots, monitoring agents). No single party
owns the crew's data. Membership is by invitation but porous
over time. This is a generalisation of Stoop's neighbourhood
case, with two twists: **shared deliverables** and **mixed
trust**.

**The recommendation: Crew = closed group + crew pod + crew
config.** Three layers that compose existing primitives.

1. **Crew group** — exactly the Stoop pattern: a group key-pair,
   an admin (or co-admins), `GroupProof` per member. The same
   closed-group governance, invitation flow, role-aware
   permissions. The relay's `acceptedGroups` mechanism gates
   messaging at the network layer.
2. **Crew pod** — a Solid pod that holds crew-canonical state
   (tasks, audit log, deliverables index, crew config). Three
   ownership patterns, **per-crew pickable**:
   - **(a) Admin's pod hosts the crew**, with members granted
     role-scoped ACL access. Cheapest; works for households,
     small teams, hobby OSS crews.
   - **(b) Dedicated crew pod** (a separate WebID + pod
     provisioned for the crew itself, multi-admin keys). Right
     answer for orgs with formal SLAs or when admins might leave.
   - **(c) Federated** — every member's pod holds *their* tasks;
     the crew pod holds an index. Useful when contributors don't
     trust the crew admin to host their data. *V2 — depends on
     pod-client federated-reader being binding.*
3. **Crew config** — `<crew-pod>/crews/<id>/config.json` carries
   the crew's name, member webids, role table (standard 5 +
   custom), skill vocabulary, cadences, DoD defaults, deliverable
   schema. Loaded once via `MemberMap.fromPodConfig` and
   referenced by every skill handler.

**Cross-crew members.** A person can be a `member` of crew A and
an `external-volunteer` of crew B simultaneously — **role
composition is per-crew, not per-identity**. The agent carries N
proofs in its vault; the app surfaces a crew switcher.

**Mixed-trust UX.** The role-policy gate is the existing seam:
`external-volunteer` can claim only tagged tasks, can't see
private tasks, can't reassign others. The "Crew + members"
screen surfaces a member chip per person/agent: `admin` /
`coordinator` / `member` / `observer` / `external-volunteer` /
`device-agent`. For neighbourhood-maintenance crews with mixed
companies + volunteers, V1 also surfaces a small
**`paid-pro` vs `volunteer` badge** on the chip — driven by an
optional `compensated: bool` flag on the role-assignment record
(defaults to `false`). Display-only in V1; unlocks future
invoicing flows in V2 without schema churn.

**Per-field split within the crew pod.** The README correctly
notes that "where does the ledger live" doesn't pick one model
per group, but a per-field split within one. V1 lands that:

- The **crew pod** (whichever pattern, a/b/c) holds: tasks (open
  + closed), audit log, deliverables index, crew config, role
  assignments.
- **Member pods** hold: that member's `skills.json`, posture
  file (`posture/<crewId>.json`), calendar
  (`calendar/*.ics`), any deliverables they author (referenced
  from the crew pod by URL).
- **Merge contract per field** is locked per Q-H4.1: LWW for body
  fields, CAS for `assignee`, append-only for the audit log +
  `reviewLog`.

**Crew lifecycle.** Crews are born, run, and *end*. V1 ships:

- **Create** — six-question wizard (mirrors Stoop's group-
  creation wizard): name, kind (`household | project | team |
  friends | maintenance`), pod-ownership choice (a/b above),
  default role assignments, default DoD policy, archive policy.
- **Pause** — admin marks the crew paused; new tasks blocked;
  existing tasks remain claimable but no fresh nudges fire.
- **Archive** — admin moves the entire ledger to `closed/yyyy-mm/`
  on the crew pod; member agents stop subscribing; relay drops
  the group from `acceptedGroups`. Reversible.

**Compatibility with SDK + substrates.** No new SDK primitives.
Composes `core.GroupManager` + `pod-client` + `item-store` +
`identity-resolver` + (optionally) `notifier`.

### Input 4 — Definition of Done + approver

**The ask, restated.** Every task has a DoD. Completion either
delivers a result (a tangible artifact reference) or is marked
complete textually. The task author chooses whether the doer
(assignee) or the author signs off — i.e. self-mark vs.
approver-required.

**The recommendation: lifecycle extension on `item-store`.**

**New states + transitions:**

```
            ┌──────────┐   claim    ┌──────────┐   submit   ┌──────────┐
   add ──→  │   open   │ ─────────→ │ claimed  │ ─────────→ │submitted │
            └──────────┘            └──────────┘            └──────────┘
                ▲                       ▲                    │       │
                │                       │            approve │       │ reject
                │                       │ release/revoke     │       │
                │                       └────────────────────┘       │
                │                                                    ▼
                │                              ┌──────────┐    ┌──────────┐
                └─── reopen (admin/coord) ─────│ complete │    │ rejected │ ──→ back to claimed
                                               └──────────┘    └──────────┘
```

- **`open` → `claimed`** unchanged.
- **`claimed` → `submitted`** new. Assignee marks it ready for
  review, optionally attaches a deliverable.
- **`submitted` → `complete`** new. Approver (creator or
  designated webid) signs off. **If `approval: 'self-mark'`,
  this transition is automatic** — i.e. the legacy V0 path.
- **`submitted` → `rejected` → `claimed`** new. Approver pushes
  back with a comment; assignee picks it up again.
- **`claimed` → `open` via `revoke`** new (master-only — see
  *One master per task* below). Distinct from `release` (which
  the assignee triggers) and `reassign` (master moves it
  directly to a named webid; V0 already had this).
- Existing `cancel` / `reopen` transitions unchanged.

**The DAG dep-release moment is `complete`** — *not* `submitted`.
Dependents stay `waiting` until approval lands. This preserves
the completion-triggers-the-next-layer property the README
identifies as the central distinction from Stoop. For
`approval: 'self-mark'` tasks, the transition is instantaneous so
dependents become `ready` the moment the assignee marks it; for
approver modes, dependents wait on the approver. `computeStatus`
is unchanged — the `ready` / `waiting` / `blocked` predicate
already keys off `complete`.

**Schema additions on the item:**

```ts
type Item = {
  ...                                    // existing fields unchanged
  definitionOfDone?: string;             // free text, optional checklist (markdown)
  approval?:        'self-mark'          // legacy default
                  | 'creator'            // task author approves
                  | `webid:${string}`;   // designated third-party approver
  deliverable?:     {
    kind: 'pod-resource' | 'url' | 'note';
    ref:  string;                        // pod URI, https URL, or inline text
    submittedAt?: number;
  };
  reviewLog?: Array<{
    at: number; by: webid;
    decision: 'submit' | 'approve' | 'reject' | 'revoke';
    note?: string;
  }>;
  master?:        webid;                 // defaults to addedBy; see "One master per task"
  parentTaskId?:  string;                // for sub-tasks; see "Sub-tasks by accepter"
};
```

**Role-policy implications** (extends the standard table):

| Role | Can submit own | Can approve | Can reject | Can revoke (master) | Can override approval mode |
|---|---|---|---|---|---|
| `admin` | ✓ | ✓ (any) | ✓ (any) | ✓ (any) | ✓ |
| `coordinator` | ✓ | ✓ (any) | ✓ (any) | ✓ (any) | ✓ |
| `member` | ✓ (own assignment) | ✓ (when designated) | ✓ (when designated) | ✓ (own tasks where they are master) | ✗ |
| `observer` | ✗ | ✗ | ✗ | ✗ | ✗ |
| `external-volunteer` | ✓ (own assignment, tagged-only) | ✗ | ✗ | ✓ (own tasks where they are master) | ✗ |

**Notifier wiring** (additions on top of the existing
deadline-reminder pattern):

- On `submitted`, schedule an **approver prompt** to the
  designated approver via the in-app inbox + (when push goes
  live) a push notification.
- On `rejected`, schedule a **reopen nudge** to the assignee
  with the reject note attached.
- On `submitted` *without* an approver tap within N days
  (default 7), escalate to crew admins.

**Where it lives.**

- Schema additions and new states on `item-store` (V1 substrate
  extension). **Rule of two** is satisfied if Stoop's
  "mark-as-done" lend-flow ever needs an approval step (lender
  confirms the borrower returned it intact); Household chore
  tracking is a third candidate consumer.
- Approver prompts in `notifier` — additive.
- New skills in `apps/tasks-v0/src/skills/`: `submitTask`,
  `approveTask`, `rejectTask`, `setApprovalMode`, `revoke`,
  `addSubtask`. Existing `claimTask` / `completeTask` keep
  working in `self-mark` mode.

**Compatibility.** Existing V0 ledger keeps working; tasks
without `approval` default to `self-mark` and skip the new
states entirely. Pod schema gains a few fields and a
`reviewLog` per-item subdocument — same merge contract as the
existing audit log.

### Input 5 — Issuer notifications + in-app inbox

**The need.** The task issuer wants to know:

- *"This task missed its deadline and is still not done."*
- *"This task has been completed."*
- *(Already in scope for approver-mode tasks)* *"Someone
  submitted; you need to review."*

**The recommendation: in-app inbox first, push later.** the author's
question — "in-app first because no Firebase yet, what do you
think?" — is exactly the right call. Three reasons:

1. **No production push channel yet.** The relay-side push send-
   half ships, but a production-grade APNs/FCM provisioning is
   not yet in place; Firebase config is parked. Shipping V1 with
   only push would block the V1 release on infra that isn't ours
   to schedule. In-app inbox doesn't depend on it.
2. **Calmness.** Stoop's pitch is "opens when you need it;
   doesn't buzz at you". Tasks should match — push should be
   opt-in, not the default. An in-app inbox that says *"3 things
   need your attention"* when the user opens the app is a better
   default than an alert that interrupts dinner.
3. **Easy upgrade path.** The notifier already supports
   per-recipient channel selection. Wiring a `PushChannel`
   alongside the `InAppInboxChannel` is additive — when the
   relay-side push is production-ready, flip the user's
   per-event preference and push starts working.

**Mechanism.**

- New `InAppInboxChannel` (a `MessagingBridge` implementation in
  `@canopy/notifier`): writes notifications to a per-user
  inbox item-store on the user's pod. The Tasks UI subscribes to
  inbox events and surfaces a badge on the app icon ("3 new").
- New notifier jobs in the Tasks app:
  - `scheduleOnce` at `dueAt` keyed `missed-deadline-${itemId}`
    — fires only if status is still `open | claimed | submitted`
    when the moment arrives.
  - `subscribe` to `item-completed` → notify
    `item.master ?? item.addedBy` with a *"completed"* entry.
  - `subscribe` to `item-submitted` → notify the approver
    (designated approver or, fallback, the master) with a
    *"needs your review"* entry.
- Per-user / per-crew preferences (settings screen): which
  events go to inbox vs. push (when push lands) vs. silent.

**Compatibility with SDK + substrates.** The `InAppInboxChannel`
is a small additive substrate change. Same shape as the existing
`PushChannel` and `NoopChannel`. The inbox-storage pattern
(`<user-pod>/inbox/<id>.json`) reuses `item-store` directly —
the inbox *is* an item-store collection of `kind: 'notification'`
items.

**Cold-boot / offline-first storage.** The inbox must be readable
*before* a pod is connected (or in local-only mode). V1
**promotes Stoop's flagged `local-store` substrate** (rule of
two satisfied: Stoop V2 + Tasks V1):

- New package `@canopy/local-store` lifted from Stoop's
  `apps/stoop/src/lib/CachingDataSource.js` and
  `apps/stoop/src/lib/SyncCadence.js`.
- Wraps any `core.DataSource` with a local cache + foreground-
  only sync.
- Tasks V1 wraps the inbox `ItemStore` with `CachingDataSource`
  → cold-boot reads return immediately from cache; pod sync
  happens on foreground.
- Stoop's `RotationScheduler` substrate-candidate joins the same
  package when V1.5 push lands (already flagged in Stoop's
  candidate list as "bundle into the same `@canopy/local-store`").
- **Recommended package contents at promotion time**:
  `CachingDataSource`, `SyncCadence`, eventually
  `RotationScheduler`. The `InAppInboxChannel` itself stays in
  `@canopy/notifier` (it's a notifier channel, not a storage
  primitive) but its default storage uses `local-store`.

### Input 6 — Sub-tasks by the accepter

**The ask.** Whoever claims a task should be able to spawn
**dependency-subtasks** under it, optionally assigned to other
crew members or device-agents. The original parent task can't
complete until all spawned sub-tasks complete.

**The recommendation: a new `parentTaskId` field + an `addSubtask`
skill, gated to claimers + masters.**

**Schema:**

- `parentTaskId?: string` on `Item`. When present, this task
  is a sub-task of the named parent.
- The parent task automatically gets `dependencies: [...,
  childId]` added (via the same CAS write the sub-task creates).

**Skill:**

```ts
// addSubtask: claimer-of-parent or parent's master can call it
await tasks.addSubtask(parentTaskId, {
  text:           "Order the planks",
  requiredSkills: ["procurement"],
  dueAt:          ...,
  approval:       "self-mark",       // child inherits parent default if omitted
  assignee:       maybeWebid,        // optional; otherwise broadcast via skill-match
});
```

**Master semantics on sub-tasks.** The default `master` for a
sub-task is the **claimer who spawned it** (i.e. the parent
task's assignee at spawn time). The parent task's master keeps
oversight via the dep edge: they can see all descendants in the
DAG view, and they can `revoke` the parent (which cascades a
prompt to assignees of orphaned sub-tasks: *"the parent was
revoked; keep, reassign, or cancel?"*).

**Why claimer-spawns + claimer-as-master.** This is the natural
delegation pattern: *"I took this task; to deliver it I need
two helper tasks."* If the parent's original issuer were the
master of every grandchild, every claimer would have to ask
permission to break work down — which kills delegation. The
parent's issuer keeps the dep-graph view, but the day-to-day
ownership of sub-tasks rests with the spawner.

**Hierarchy is the dependency graph, not user rank.** Per
the author's preference. The depth of "who can revoke whose work" is
read off the DAG: master of task X can revoke X (cascading to
its children); master of child Y can revoke Y; the grandparent's
master cannot revoke Y directly. **No role-rank-based
hierarchy** beyond the standard policy gate. The `admin` /
`coordinator` roles have crew-wide override (they can revoke any
task in their crew); below that, hierarchy is the dep edge.

**Cycle detection.** The existing `detectCycle` (write-time)
covers this — adding a sub-task of an ancestor would create a
cycle and is rejected.

**UI affordances.** The DAG view (workspace screen 5) shows
parent-child indentation; the task detail view shows
*"Sub-tasks (2 of 3 complete)"* with a *"+ add sub-task"*
button visible to the claimer + master.

**Depth policy (admin-tunable, no hard cap).** No fixed maximum
depth. Each crew config carries
`subtasks.adminApprovalDepth: number` (default `3`) — when
spawning a sub-task that would land at depth > `adminApprovalDepth`,
the system queues an admin-approval request instead of
auto-creating it. Admin sees it in their review queue and
either approves (the sub-task is created) or declines (with an
optional comment back to the spawner). This is a graceful
escalation: shallow trees are frictionless; deep trees ask for
attention without forbidding them outright. **Recommendation
for default: 3** — most practical task breakdowns don't go
deeper than parent → child → grandchild; anything deeper is
worth a moment of admin sanity-check.

### Input 7 — One master per task + unassign

**The ask.** *"It should be possible to unassign people from a
task if they took the task from you (so one master per task —
maybe add a hierarchy later? — based on task dependencies, not
on user ranks or anything like that)."*

**The recommendation: explicit `master` field; new `revoke` skill;
hierarchy via dep graph.**

- **`master: webid`** field on the item. Defaults to `addedBy`
  on creation; defaults to the spawner on sub-task creation;
  admins / coordinators can overwrite (rarely needed).
- **`revoke(taskId, reason)` skill** — master-only. **Reason is
  mandatory** (free text, ≥ 1 char). Sends the task back to
  `open` (clearing `assignee`, `claimedAt`); does not delete the
  task or its history. Logged in `reviewLog`.
- **`appeal(taskId)` skill** — assignee-of-the-revoked-task only,
  available for 7 days after a revoke. Opens a `chat-agent`
  thread between the previous assignee and the master, pre-
  loaded with the revoke reason. No decision logic in the skill
  itself — it's just a channel for the conversation. The master
  can call `reassign(taskId, previousAssigneeWebid)` from the
  thread if they want to undo.
- **Notifier wiring** — on `revoke`, the inbox gets an entry
  *"Anna revoked your assignment on 'Order paint'. Reason:
  'Deadline pushed forward; doing it myself'. [Appeal]
  [Dismiss]"*. The Appeal button calls the `appeal` skill.
- **Distinct from `release`** (assignee voluntarily gives the
  task back) and **`reassign`** (master moves the task to a
  named webid in one step; V0 already has this).
- **Dependents on a revoked task** — the revoke just changes
  who the assignee is; it doesn't change the task's status to
  anyone but the previous assignee. Children (sub-tasks)
  spawned by the previous assignee stay assigned to *their*
  spawned-master (which was the previous assignee) — the new
  assignee can choose to keep working with them, ask them to
  reassign, or revoke their own children.

**This is the simplest answer to "one master".** The task has
exactly one master at any time (default = creator); the master
is the only person other than admin/coord who can yank the
assignee. Everyone else's role is governed by the existing
role-policy table.

**Forward-compat for hierarchy.** The user explicitly asked for
hierarchy via the dep graph, not user rank. The DAG is already
there; no new primitive. When sub-tasks proliferate, the UI's
DAG view + ancestorChain helper is the navigation surface.

## Local-only operation (hard rule for V1)

The whole app must be usable **without a Solid pod connection**.
The boot path, all skills, the inbox, the calendar reader, and
the workspace UI must work against local-only storage. Pod sync
is an attach-when-configured feature, not a precondition.

**Why it's load-bearing.**

- The single most error-prone UX path in the project today is
  pod sign-in (cap-tokens, OIDC redirects, Inrupt issuer
  selection). Forcing the user through it before they can add
  one task is a 5-minute friction wall on first launch.
- Households and friend groups in particular don't need cross-
  device sync to get value out of the app on day 1.
- Local-only is the test-mode default for development.
- Stoop's `CachingDataSource` already proves the pattern works.

**Mechanism.**

- `core.DataSource` is the pluggable seam. Local-mode boots
  with a `LocalFileSource` (writes to `~/.tasks/` on
  desktop / Application Support on macOS / `expo-file-system`
  on RN). Pod-mode wraps `pod-client.PodClient` in the same
  DataSource interface.
- `CachingDataSource` (lifted into `@canopy/local-store`)
  wraps either, making local-mode and "pod-mode-while-offline"
  the same code path.
- Calendar reader has the same dual-mode (`apps/tasks-v0/src/calendar/iCalReader.js`
  reads from a local directory or a pod container; the caller
  picks).

**README rule (added to `apps/tasks-v0/README.md` per the author's
note):** *"Tasks works locally without a Solid pod. Connecting
a pod adds cross-device sync; it is never required."*

## Pod-data sharing — caution principles

Several V1 stories reach into a *member's own pod* — calendar
files, skill profiles, deliverables, posture files. This is the
right design (the user controls their data) but it deserves a
discipline so we don't accidentally over-share:

1. **Default opt-in per crew, not per-app.** Granting the Tasks
   app permission once doesn't grant permission to share data
   with every crew the user joins.
2. **The smallest derivative wins.** `getFreeBusy` returns busy
   intervals, *not* events; the deliverable manifest carries a
   pod URL the recipient must dereference (so revocation is one
   ACL change, not "redistribute the data").
3. **Audit the cross-pod reads.** Each cross-pod read goes
   through the role-policy gate; the audit log records who-read-
   what.
4. **Ask the user before adding new flows.** This is a process
   rule for the design discipline: any new feature in a future
   coding plan that introduces a *new* cross-pod read (beyond the
   ones listed in this doc) gets explicitly flagged and the author
   signs off before it ships.

The V1 cross-pod flows currently listed:

- **Skill + posture read** for skill-match and the assignee
  picker (member pod → crew).
- **Deliverable read** when an approver opens a submission
  (member pod → crew).

Both are scoped to crew membership and gated by ACPs on the
member's pod. **Calendar data does NOT cross pods** in V1 (locked
2026-05-07) — calendar matching is local to the user; nothing
about my agenda reaches anyone else. Anything beyond these for
V1+ requires explicit sign-off — the **caution principles**
belong in the project README. *Updates to other app READMEs to
inherit this principle are flagged in the cross-app section
below.*

**Net conclusion.** Tasks V1 triggers:

- **Two substrate extensions** — `item-store` (DoD lifecycle +
  sub-tasks); `notifier` (`InAppInboxChannel`).
- **One new substrate package** — `@canopy/local-store`
  (lifted from Stoop's `CachingDataSource` + `SyncCadence`,
  rule of two satisfied; required for the local-only mode rule).
- **One app-level helper** — local calendar reader (no
  substrate; pure helper).
- **One handoff to another app** — calendar source connectors
  go to `apps/import-bridge-v0`.
- **One V1.5 promotion** — `PushPolicy` from Stoop into
  `notifier` when push flips on.

User stories below cover the V1 scope (without push); V1.5 adds
push notifications + chat-bot bridge + custom-role UI without
shape-changing the data model.

## Recommended high-level design

### Agent topology

- **One agent per user, per crew.** The agent's identity is
  anchored on the user's pod; its `GroupProof` for *this crew's
  group* lives in its vault.
- **Crew switcher** lets a user belong to N crews; UI swaps
  which agent is active. Mirrors Stoop's group switcher.
- **Device-agents are first-class** — a build bot, a 3D printer,
  a monitoring agent each carries its own webid + posture file +
  `GroupProof`. `humanInTheLoop: false` flag drives the UI label.
- **Agents = SDK Agent class.** Transport selection (mDNS / BLE /
  relay) is automatic via `RoutingStrategy`.
- **Aligns with Agent Hub direction** — per-app-agent today,
  hub-attachable later.

### Substrate composition (for reference)

```
apps/tasks-v0  (UI + crew workspace + onboarding + V1 glue)
   │
   ├── @canopy/skill-match       (skill-tagged dispatch within a crew)
   ├── @canopy/item-store        (tasks + DoD + approval + sub-tasks + master)
   ├── @canopy/identity-resolver (per-crew member map + reveal state)
   ├── @canopy/agent-ui          (REST + SSE + mountLocalUi)
   ├── @canopy/notifier          (deadline + approver-prompt + InAppInboxChannel)
   ├── @canopy/chat-agent        (optional task threads)
   ├── apps/tasks-v0/src/calendar  (Solid-iCal read adapter — substrate-candidate)
   └── apps/folio/src/imports/calendar  (calendar-to-pod sync)
        │
        └── @canopy/core + @canopy/relay + @canopy/react-native (SDK)
```

## Chat-bot access (committed to V1.5 — most plumbing already ships)

**The ask.** A Telegram (or own-chat) bot that can query the
task ledger and act on it: *"what's open?"*, *"what are my
sub-tasks?"*, *"what blocks task X?"*, *"approve task Y"*.

**Prior art already in the repo.** `@canopy/chat-agent` (L1c)
ships `TelegramBridge` + `InMemoryBridge` + `MessagingBridge`
as substrates. `apps/household/src/bridges/TelegramBridge.js`
is a re-export shim around the substrate; the bridge itself
handles addressed-only filtering, webhook + long-polling,
inline-button-tap synthesis, and graceful shutdown. **No bridge
work needed for Tasks** — just import.

**V1.5 scope (revised down to ~2-3 days).**

- Import `TelegramBridge` from `@canopy/chat-agent`; wire it
  to the Tasks agent in a new `apps/tasks-v0/src/bot/` module.
- Register *bot-bound* skills in the Tasks agent: `bot.listOpen`,
  `bot.whatBlocks`, `bot.listMine`, `bot.markComplete`,
  `bot.approve`, `bot.appeal`. Each wraps the equivalent
  UI skill with a chat-friendly text + button response.
- Capability-token-bound bot agent (lift Stoop V2's substrate-
  candidate pattern: bot is an agent under the user's root
  identity, with a scoped cap-token specific to that bot
  binding). If Stoop V2's pattern hasn't lifted yet, Tasks can
  ship the same shape inline and become the second consumer.
- Audit-log entries distinguish *"Anna via web"* vs *"Anna via
  Telegram bot"* so the action trail is honest.

**V1 design implications (so V1.5 lands cleanly).**

- Every skill the UI calls is reachable through `agent.call`
  — no UI-only paths. Already the case in V0; don't regress.
- Per-task state lives on the pod / item-store, never in the
  UI session. Already the case.
- The future bot is just another agent in the crew; no special
  routing.

**Note for future me / future critique passes:** before
estimating "new" features, grep the repo for prior art. I
costed this at 5 days originally; the actual number is closer
to 2-3 because the bridge substrate already ships.

## User stories (V1)

Each story is keyed to the SDK / substrate primitive that
satisfies it. Names (Anna / Bob / the author / Carol) follow Stoop's
convention.

### Onboarding + crew creation

1. **As Anna I start Tasks for the first time and have a working
   profile within 3 minutes.** Default Solid pod issuer pre-
   selected (mirroring Stoop). SDK: `core.Bootstrap`.
2. **As Anna I scan a QR code from a fellow crew member to join
   "OSS Tools NL".** Same flow as Stoop. SDK: `GroupManager.redeemInvite`.
3. **As Anna I create a new crew "Community garden Oosterpoort"
   via a short wizard** (name, kind, pod-ownership choice, default
   approver, archive policy). The wizard chooses between
   pod-ownership patterns (a / b) with a plain-language
   explanation per option.

### Tasks + skills + deadlines

4. **As the author I create a task "Deliver upholstery" with required
   skill `sewing` and a deadline of Friday.** The composer
   suggests `sewing` from the crew skill vocabulary. SDK:
   `addTask` + `skill-match` broadcast.
5. **As Bob I receive an in-app inbox entry that a sewing task is
   available.** Substrate: `notifier.InAppInboxChannel`. *Push
   wakes when V1.5 enables the relay-side push.*
6. **As Bob I see my calendar conflicts on the deadline date when
   I am about to claim** — the UI calls my `getFreeBusy` against
   the `dueAt` window. Substrate: app-level Calendar adapter.
7. **As the author I see, during assignment, which candidates are free
   on the deadline.** Same skill, called against multiple webids
   in the crew member map.

### Decentralised crews + cross-org

8. **As Carol — `external-volunteer` in the garden crew and
   `admin` in the OSS-tools crew — I switch between my crews with
   one tap.** Crew switcher; same UX pattern as Stoop's group
   switcher.
9. **As the garden admin I invite a paid gardener as
   `external-volunteer`.** The gardener sees only tagged tasks
   meant for them; my private notes stay hidden. Substrate:
   role-policy gate + `visibility: 'role:external-volunteer'`.
10. **As the OSS-crew admin I use a dedicated crew pod (option
    b) so the crew is not tied to my personal pod.** SDK:
    `pod-client` configured with the crew-pod URI in the crew
    config.

### Definition of Done + approval

11. **As the author I create a task with DoD "the fence is painted and
    3 photos are uploaded" and set myself as approver.** Schema:
    `definitionOfDone` + `approval: 'creator'`.
12. **As Bob I do the task and mark it submitted with a link to
    the photos in my pod.** Lifecycle: `claimed → submitted`;
    `deliverable: {kind: 'pod-resource', ref: ...}`. *Note: my
    pod is the source of truth for the photos; the author dereferences
    via the link, his read is gated by the ACP I set on my pod.
    See "Pod-data sharing — caution principles".*
13. **As the author I get an in-app inbox entry: "Bob has submitted
    'Paint the fence' for your approval".** Substrate: notifier
    + `InAppInboxChannel`.
14. **As the author I approve, or reject with a comment "the side
    photo is missing".** `approveTask` / `rejectTask`; on reject
    the task returns to `claimed` with a `reviewLog` entry.
15. **As Bob I see on my work board which tasks are awaiting
    approval or were rejected.** `listMine` with status filter.
16. **As the admin of a large crew I mark a new task as
    `approval: 'self-mark'` so trivial work doesn't pile up in
    my approver inbox.** Per-task override of the crew default.

### Sub-tasks + master + revoke

17. **As Bob — assignee of "Build the new shed" — I spawn a
    sub-task "Order planks" assigned to Carol.** New skill:
    `addSubtask`; substrate change: `parentTaskId` field on the
    item; Bob is the master of the new sub-task (parent's
    issuer keeps oversight via the dep edge).
18. **As Anna — issuer and master of "Build the new shed" — I
    revoke Bob's assignment because he's overcommitted.** New
    skill: `revoke`; the task returns to `open`. Bob's spawned
    sub-task (Order planks → Carol) is unaffected; Carol can
    keep working unless I revoke that one too.
19. **As Bob I see in the DAG view that my parent task has been
    revoked, and I'm prompted to keep / reassign / cancel my
    spawned sub-tasks.** UI affordance, not a new skill.
20. **As the author I drill down through 3 levels of sub-tasks in the
    DAG view to see what's blocking the deliverable.** UI
    affordance over `dag.computeStatus` + `parentTaskId`.

### Devices + automation

21. **As 3D-printer-owner Anne I register my printer as a crew
    member with `posture['3d-print']: 'always'`.** The printer
    auto-claims and submits a deliverable
    (`{kind: 'pod-resource', ref: 'pod://.../prints/2026-05-09.gcode'}`).
    `approval: 'self-mark'` is the default for `humanInTheLoop:
    false` agents — otherwise tasks wait forever.

### Operational + multi-tenant

22. **As the admin I pause the crew during the summer break.** No
    new tasks, no nudges; existing tasks stay claimable.
23. **As the admin I archive the crew once the work is done.**
    Ledger goes to `closed/yyyy-mm/`; relay drops the group from
    `acceptedGroups`.
24. **As a volunteer I leave a crew — my submitted deliverables
    stay linked (they live on my pod, gated by my ACP), my open
    tasks are released.** `leaveCrew` (the analogue of Stoop's
    `leaveGroup`).

### Calendar + availability

25. **As a user I connect my Google calendar via Folio's calendar
    bridge, which writes iCal files to my pod.** Folio holds the
    OAuth token; I review which crews have calendar-read
    permission. *Cross-pod read; opt-in per crew per "Pod-data
    sharing — caution principles".*
26. **As a user I see in the crew settings that I am currently
    sharing `getFreeBusy` with this crew, and I can revoke
    without disconnecting Folio's sync.** Per-crew skill-posture:
    `getFreeBusy: 'never'` for crews where I don't want to share.

### Issuer notifications + in-app inbox

27. **As the author I get an inbox entry "Deadline missed: Order paint
    — assigned to Carol — was due 2 days ago" and I can either
    re-broadcast, revoke, or open a thread with Carol.**
    `notifier.scheduleOnce` keyed on `dueAt`.
28. **As the author I get an inbox entry "Task completed: Paint the
    fence" when Bob submits and I have `approval: 'self-mark'`,
    or after I approve in `creator` mode.** Subscribed to
    `item-completed`.
29. **As an admin / coordinator I see issuer-fan-out: an inbox
    entry per task I created that misses its deadline, batched
    into a daily digest if more than 5 fire in a day.** Notifier
    `PushPolicy`-style batching; behaviourally identical when
    `InAppInboxChannel` is the carrier.

## Observability + admin-tunable cadence (V1)

Per critique #13: V1 ships per-crew metrics so cadence
defaults can be tuned from real usage instead of guessed from
spec.

**Settings architecture:**

- Each crew config carries a `cadence` block with admin-defined
  defaults: deadline-reminder lead time, missed-deadline notice
  timing, approver-prompt cadence, sub-task admin-approval
  threshold, push-vs-inbox routing per event type.
- Each user profile carries a `cadence-overrides` block —
  per-event opt-out / quieter / louder, scoped to that user.
- Resolution at notifier-call time: user override (if set) > crew
  default > app baseline.

**Metrics surfaced in the crew settings → "stats" tab** (V1, not
V1.5 — admins need the data to tune the cadence at all):

- Tasks: added / claimed / completed / revoked / rejected /
  approved / missed-deadline (counts, weekly).
- Latencies: time-to-claim, time-from-submit-to-approval,
  notification-arrival-to-user-action (median + p90).
- Approval friction: rejection rate, average review-cycle count
  per task.
- Notification volume: per-user sent count, per-event-type
  share, opt-out frequency.

Locally aggregated; opt-in to share with admin (per the pod-
data-sharing caution principles). Defaults adjust at V1.5/V2
based on what real crews show.

## Phased plan

### Phase 1 — Tasks V1 (crew envelope + DoD + sub-tasks + local calendar + in-app inbox)

- Adopt `apps/tasks-v0` as the engine; introduce **Crew**
  envelope (config loader + group binding + per-crew skill
  vocabulary).
- Add `submitted` + `rejected` states + DoD / approval /
  deliverable / `master` / `parentTaskId` fields on
  `item-store`; new skills `submitTask` / `approveTask` /
  `rejectTask` / `setApprovalMode` / `revoke` (mandatory
  reason) / `appeal` / `addSubtask`.
- Standard 5-role policy only (custom roles deferred to V1.1).
- Sub-task admin-approval threshold (`subtasks.adminApprovalDepth`,
  default 3) — admin queue handles spawn requests beyond it.
- Build the workspace UI screens (workspace, add task, my work,
  review queue, DAG, crew + settings + stats, inbox).
- **Promote `@canopy/local-store` substrate** (lift Stoop's
  `CachingDataSource` + `SyncCadence`); wrap inbox storage and
  the canonical task data source through it.
- Add `InAppInboxChannel` to `@canopy/notifier` (additive);
  wire issuer-notification jobs (missed-deadline, completed,
  submitted, revoked-with-appeal) through it.
- App-internal local calendar reader (`apps/tasks-v0/src/calendar/iCalReader.js`):
  reads `*.ics` from pod-mode container OR local-mode directory;
  conflict-only view in claim flow. **No network skill, no
  cross-pod read.**
- Mockup `.ics` fixtures under `test/fixtures/calendar/` for
  tests + dev-mode pod-mock.
- Local-only mode wired end-to-end: app boots without pod, all
  features work, pod attaches when configured.
- Skill-import-from-pod prefilled-form helper (canonical
  `<user-pod>/profile/skills.json`).
- Crew archive + pause flows.
- Per-crew cadence config + observability stats tab.
- Localisation: ship `nl` + `en` per project convention.
- Closed-beta privacy notice (mirror Stoop's content; explicitly
  call out that calendar data stays on the user's device).

### Phase 1.5 — Push wake-up + chat-bot bridge

- Promote Stoop's `PushPolicy` substrate-candidate into
  `@canopy/notifier`.
- Wire push as a second channel for issuer / approver
  notifications, alongside the in-app inbox. Per-event toggle
  in settings (driven by admin defaults + user overrides).
- Requires the relay-side push provisioning to land first.
- **Chat-bot bridge** (~5 days): TelegramBridge from household
  app + capability-token-bound bot agent + `bot.*` skill set
  (`bot.listOpen`, `bot.whatBlocks`, `bot.listMine`,
  `bot.markComplete`, `bot.approve`).
- **Custom-role UI** activation (Q-H4.7 (c) extension) —
  small management screen.

### Phase 2 — Multi-tenant hardening

- Crew pod ownership pattern (b): provisioning helper for a
  dedicated crew pod.
- Crew pod ownership pattern (c): federated reads across member
  pods (depends on `pod-client` federated-reader being binding).
- Bulk operations (reassign-on-holiday, "all my open tasks").
- Recurring tasks (Q-H4.9 unpark) — RRULE-based; integrates
  naturally with the calendar adapter.
- Multi-claim / co-assignment for tasks that need a pair.

### Phase 3 — V2 features

- Auto-scheduling planner.
- Cross-crew dashboard.
- Real-time collaboration on a deliverable doc (waits for
  project #1 OSS-doc-tool integration).
- Compensated-role flag + invoicing primitives.
- Cryptographic anonymity.
- Cross-tool calendar sync (write-side: emit `VEVENT` to the
  pod when a task is scheduled).
- Cross-member availability hints if a future opt-in flow makes
  the privacy trade-off acceptable.

## Open questions (status as of 2026-05-07 after review)

Resolved during the design review:

- ~~**Naming of the envelope.**~~ **Resolved: keep "Tasks" as
  the app name.** The "Crew" working term stays for the
  *envelope concept* (a tasks group), but the app is plainly
  called Tasks. Tasks is the generalised app; specialised
  apps (Stoop already, Household already, future ones) build
  on the same primitives.
- ~~**Calendar adapter scope.**~~ **Resolved: local matching
  only; no network freebusy.** Calendar reads stay on the
  user's device; conflicts are shown to the claimer only.
- ~~**Calendar sync — where does it live?**~~ **Resolved:
  `apps/import-bridge-v0`** as another source connector. Folio
  just reads `*.ics` from the pod when it needs to display.
- ~~**Listener vs. poll for non-Google/non-Outlook
  calendars.**~~ **Tracked under import-bridge** —
  out of Tasks V1 scope.
- ~~**Calendar privacy default.**~~ **Resolved: not a question
  any more — calendar data doesn't leave the user's device.**
- ~~**Calendar adapter dependencies.**~~ **Resolved: use
  `ical.js`.**
- ~~**Sub-task explosion.**~~ **Resolved: no hard depth cap.
  Per-crew `subtasks.adminApprovalDepth` (default 3); spawning
  past it queues an admin-approval request.**
- ~~**Custom roles in V1.**~~ **Resolved: deferred to V1.1.**
  V1 ships standard 5 roles only. UI for custom roles + the
  Q-H4.7 (c) extension path lands in V1.1.
- ~~**Chat-bot access.**~~ **Resolved: V1.5 with budgeted
  scope (~5 days)**, not "forward-compat" hand-waving.
- ~~**Observability cadence.**~~ **Resolved: admin defaults
  per crew; user overrides per event.** Stats tab in crew
  settings ships in V1 so admins have data to tune from.
- ~~**Local-only operation.**~~ **Resolved: hard rule for
  V1.** App must work fully without a pod connection.

Still open:

1. **Crew pod ownership default.** V1 default — (a)
   admin-hosts-the-crew, or push the user toward (b) dedicated-
   crew-pod from the start? **Lean: (a) for V1, with "migrate
   to dedicated pod" as a one-click flow in V2.**
2. **Approver chain.** Single approver per task, or a chain?
   **Lean: single in V1; chain in V2 if a real consumer asks.**
3. **Deliverable typing.** Free-form (`{kind, ref}`) vs. crew-
   defined schemas. **Lean: free-form for V1; schema-validated
   in V2.**
4. **Crew membership porosity.** Can a crew member invite
   another, or is invitation admin-only? **Lean: admin-only in
   V1 to keep governance simple; per-crew policy switch in V2.**
5. **Cross-crew skill aggregation.** Per-crew (declared
   multiple times) or once-and-imported via the canonical
   `<user-pod>/profile/skills.json`? **Lean: once-and-imported
   with per-crew posture override** (already documented as the
   skill-import pattern; restating here for completeness).
6. **Push provisioning.** When does relay-side push become
   production-grade enough to flip the V1.5 feature flag?
   Coordination with the Stoop team. **Lean: track Stoop V2's
   push status; reuse the same provisioning.**
7. **Master override by admin/coord.** Should `admin` /
   `coordinator` be able to overwrite `master`, not just revoke?
   **Lean: yes — already implied by the role-policy table;
   surface as an explicit affordance only when needed.**
8. **In-app inbox storage location.** Per-user pod
   (`<user-pod>/inbox/`, privacy-clean) or per-crew pod
   (shared, weird)? **Lean: per-user pod — the inbox is *for*
   the user, not *about* the crew.**
9. **Default `subtasks.adminApprovalDepth`.** Recommendation
   is 3. **Open**: validate against 2-3 real use cases before
   locking; might land at 2 for households (where deep
   hierarchies are noise) or 4 for OSS projects.
