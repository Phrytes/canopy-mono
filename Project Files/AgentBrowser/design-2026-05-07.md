# Agent-SDK browser (2026-05-07, seed)

> Status: **idea-capture / seed doc**. Detailed design + roadmap
> follow once Stoop V2 ships and the platform value-proposition has
> been validated with closed-beta users.
>
> Companion to [`../AgentHub/monitoring-design-2026-05-07.md`](../AgentHub/monitoring-design-2026-05-07.md).

## Wat dit is

Een **cross-platform sandbox-runtime** specifiek voor agent-SDK-apps.
Niet "een app", maar **de schil waar agent-SDK-apps in draaien** —
zowel op desktop als mobiel. De browser:

- laadt elke JS-app die het agent-SDK-protocol spreekt,
- zet de SDK-bindings + RN-bindings al voor je klaar (één keer goed gedaan, alle apps profiteren),
- handhaaft een capability-manifest per app (geen cookies, geen ongedeclareerde externe verbindingen),
- logt alle skill / transport / pod / network events naar de [Hub-monitoring laag L1](../AgentHub/monitoring-design-2026-05-07.md),
- vraagt eerste keer om akkoord wanneer een app een externe service aanspreekt (bv. Nominatim voor geocoding).

Het uitgangspunt: **alles draait zoveel mogelijk lokaal**; data wisselt
uit met Solid + peers; externe diensten zijn opt-in en zichtbaar.

## Waarom dit cruciaal is voor het hele ecosysteem

Twee winsten in één:

1. **Veiliger** — ongetrouwde apps kunnen niet stiekem cookies opslaan, stille netwerkverbindingen openen, of pod-data lekken naar een advertentie-API. De runtime weigert het op systeemniveau, vóór de app de SDK ziet.
2. **Eenvoudiger app-development** — veel kruis-platform-logica zit al in de browser ingebouwd. Eén app-bundle draait op desktop én mobile zonder twee codebases. RN-specifieke bindings (camera, GPS, push, BLE) en web-specifieke (file upload, clipboard) worden achter dezelfde unified API gebracht.

Beide voordelen versterken elkaar: hoe meer je in de runtime stopt,
hoe makkelijker het ontwikkelen wordt — *en* hoe minder oppervlak de
app heeft om misbruik te maken.

## Wat een app krijgt (voor de ontwikkelaar)

Een app is een ZIP / tarball met:

```
my-app/
├── agent-app.json     (manifest: name, version, capabilities)
├── index.html         (UI entry point)
├── app.js             (browser-side wiring; uses the SDK + the unified bindings)
├── agent.js           (the agent factory; runs in a worker / native shell)
└── assets/            (CSS, locales, images)
```

Bij `agent-browser open ./my-app/`:
1. Browser leest het manifest. UI toont *"Deze app wil: A, B, C. Akkoord?"*.
2. Op accept: de app wordt in een sandboxed context geladen. Cookies disabled. `localStorage` alleen toegankelijk voor de eigen app-namespace. `fetch()` gefilterd via de declared external-hosts.
3. SDK-imports (`@canopy/core`, `@canopy/identity-resolver`, etc.) worden runtime-geleverd door de browser — de app hoeft ze niet te bundelen.
4. Per-platform bindings (camera/GPS/push/BLE op mobile; file/clipboard/notification op desktop) komen achter één unified API.
5. Audit events flow via Layer-1 substrate → optionally to a Hub.

## Wat een gebruiker krijgt (voor de end user)

- **Eén app om alle agentic apps te draaien**, zoals een browser voor het web.
- **Capability prompt** bij installatie: dezelfde mobile-style permission-flow.
- **Audit timeline** van alle apps in één scherm.
- **Geen reclame, geen tracking** — de browser belooft dit op runtime-niveau, niet alleen in een privacy-policy.
- **Op zowel laptop als telefoon hetzelfde**: dezelfde apps, dezelfde audit, dezelfde permissions. Identiteit en pod-data via Solid universeel.

## Architectuur — drie lagen

| Laag | Op desktop | Op mobile | Status |
|---|---|---|---|
| **Shell** | Electron of Tauri (Rust + WebView). Zelf-update, app-installer, capability-manifest UI. | Native iOS/Android shell (Swift/Kotlin) of Capacitor. Hosts JS-bundle in een sandboxed WebView. | Nieuw bouwwerk; zie effort hieronder. |
| **Runtime API** | JS-runtime (Node + WebView combinatie) waar SDK + unified bindings beschikbaar zijn. | Idem (JavaScriptCore / Hermes met dezelfde unified API). Per-platform impl van GPS/camera/push/BLE. | Composeert bestaande `@canopy/react-native` substrates. |
| **App-bundle** | ZIP / tarball met manifest + JS + assets. Ondertekend door de developer (optioneel). | Idem. | Standaard format — apps schrijven hier eenmalig tegenaan. |

**Cross-platform parity is het kernidee.** Eén app-bundle draait
identiek op beide. Dit haalt het grote nadeel van de huidige
SDK-positie weg: nu moet elk app-team React-Native-kennis bouwen om
ook mobiel te leveren. Met de browser is dat opgelost door het
platform.

## Waar dit verschilt van een gewone webbrowser

- **Geen cookies, ooit.** Standaard. Apps die state willen vasthouden gebruiken het pod-namespace, niet cookies.
- **Geen tracking pixels / advertenties.** Het netwerk-policy is whitelist (manifest-declared hosts), niet blacklist.
- **Geen JS-bundle van het web** — apps zijn lokaal-geïnstalleerd of via signed bundle van een Hub-curated catalogus.
- **Geen DOM-tracking-API's**: geen `Navigator.userAgent` shenanigans, geen WebGL fingerprinting, geen battery-status-API.
- **Wel** de SDK + unified RN-style bindings in scope — dat is de runtime-meerwaarde.

In essentie: het is een **browser voor agentic apps**, niet een browser voor het hele web.

## Effort-schatting

| Mijlpaal | Effort | Waar |
|---|---|---|
| **Desktop MVP** — Electron-shell die één Stoop-bundle laadt + manifest-prompt + network-whitelist. Geen Hub-koppeling. | 2-3 weken | bouwt op bestaande Stoop bundle |
| **Audit logging** geïntegreerd (Layer-1 substrate vereist) | 1 week extra | hangt af van de monitoring-substrate |
| **Mobile MVP** — Capacitor of native shell met dezelfde JS-runtime | 3-4 weken | grootste werk; native bindings opnieuw |
| **App-installer + signed bundle support** | 2 weken | uitbreiding |
| **Hub-koppeling** (event-stream upload + capability-violation alerts) | 1 week | hangt af van Hub-fundament |

Realistisch: **2-3 maanden** vanaf nul tot een werkbare cross-platform browser met een handvol apps. Dat is een eigen projectspoor.

## Wat dit nu nog niet doet

- **Geen multi-tab UI.** Per-app proces; volgens de mobile-style "ik open Stoop" / "ik open Folio" flow.
- **Geen platform-dependency op eigen browser-engine** — Chromium / WebKit blijven de underliers. Enige gain is in het sandbox-gedrag eromheen.
- **Geen vervanging voor de huidige web-deployments** van Stoop / Folio. Die blijven werken; de browser is een extra deployment-target.

## Open vragen

1. **Electron of Tauri?** Tauri is leaner (Rust core, system WebView) maar minder mature ecosystem. Electron is dik maar bewezen. Voor V1 desktop: waarschijnlijk Electron; Tauri later als de community volwassener is.
2. **Mobile shell**: native (Swift/Kotlin) vs Capacitor (één JS-codebase, native shell). Capacitor zou het cross-platform parity-verhaal makkelijker maken.
3. **App distributie**: Hub-curated catalog (signed bundles, reputation), peer-to-peer (gebruiker krijgt de URL van een vriend), of self-hosted (developer host hun eigen index.json met versies).
4. **Update flow**: wanneer een geinstalleerde app een nieuwe versie heeft die meer capabilities vraagt → re-prompt user. Hub kan signaleren "deze update is verdacht: van 1 host naar 30 hosts".
5. **iOS distributie**: Apple App Store laat geen "alternative app stores" toe op iOS. Dit is een blokkade voor de mobile-versie op iOS — maar Android werkt wel. Op iOS kan het via een PWA-fallback (zonder de native bindings).
6. **Verhouding tot de Hub**: kun je de browser draaien zonder Hub? **Ja** — Layer-1 audit logs blijven lokaal. De Hub is opt-in en geeft cross-app correlatie.

## Volgorde in het bredere plan

Dit is een **eigen project naast Stoop V2/V3**, niet een Stoop-feature.
De redelijke ordening:

1. Stoop V2 (Phases 23-30) ships eerst — bewijst de SDK-positie + maakt een ervaringsbron op.
2. **Layer-1 monitoring substrate** in core (zie [`monitoring-design-2026-05-07.md`](../AgentHub/monitoring-design-2026-05-07.md)) — voorwaarde voor het audit-stuk in de browser.
3. **Browser desktop MVP** — focus op één app (Stoop V2.5) tot het echt werkt.
4. **Browser mobile MVP** — Stoop draait dan op telefoon zonder dat Stoop een eigen mobile-team nodig had. Dit is de magische demo.
5. **App distribution + Hub-koppeling** — de "platform"-stap.
