### Phase 52.16 — Sharing v2 (≈5 days) ✅ **SHIPPED 2026-05-14**

Lands after 52.15. Tasks detailed in design §5 (52.16.1 – 52.16.8).

**All 8 sub-phases shipped 2026-05-14.** Final test counts: pod-client
188/190 (+37 sharing — 14 capabilities + 23 sharing), Folio 463/463
(+11 mode/ACP), Folio-mobile 79/79. `client.sharing.*` runs against
Inrupt's `universalAccess` (lazy-loaded; cleanly absent for non-Inrupt-
SDK environments). Folio's auto-share probes per-origin and prefers
ACP when supported; cap-token remains the fall-back for non-ACP pods
and offline / power-user scenarios.

**Critical-path output:**
- `client.sharing.{grant, revoke, list, capabilities}` API in
  `@canopy/pod-client` (net-new code using `@inrupt/solid-client`
  ACP primitives).
- Folio CLI/server/web/auto-share + folio-mobile ShareScreen adopt
  the new API with `--mode cap-token|acp` switch.
- `SharingUnsupportedError` + capability probe for non-ACP pods.

**Acceptance gate (from design §5):** Folio defaults to ACP-mediated
grants when the pod supports it; falls back to cap-token cleanly;
already-issued cap-tokens remain valid via the unchanged
consumer-side path.

  unblocks new sign-in UX for Tasks V1 / Household V2 and (b) it's
  smaller — quicker to ship.
- Both phases are **independent of the already-shipped Phase 52.14
  (Q-D conflict resolution)** and Phase 52.9.2 (Q-B groupMirror

## 4. Cap-token deprecation policy (explicit lock)

Cap-token cryptography (`PodCapabilityToken` + `CapabilityAuth`) **is
not deprecated**. Sound per saved memory + still load-bearing for
bot/admin scope.

What changes:

| Surface | V1 disposition (post-52.16) | Future |
|---|---|---|
| `@canopy/core.PodCapabilityToken` | **Keep** | No deprecation. |
| `@canopy/pod-client.CapabilityAuth` | **Keep** | No deprecation. |
| Folio `folio share <path>` CLI | **Keep with `--mode` switch** (default: probe; explicit `--mode cap-token` for power-user "I want a token to send out-of-band") | No deprecation. |
| Folio server `POST /share` | **Keep, returns `{mode, token? \| grantId?}`** | No deprecation. |
| Folio browser Share pane | **Default UX flips to ACP** when supported; falls back to cap-token automatically. UI labels the mode. | No deprecation of the cap-token fallback. |
| Folio `with-<webid>/` auto-share | V1: cap-token mode (today). V1.5: `acp` mode when consumer's pod supports it. | Folder convention itself is not deprecated. |
| Folio-mobile ShareScreen | Same as desktop: picks ACP when supported, cap-token otherwise. | No deprecation. |
| Tasks-mobile bot-token QR | **Out of scope** — skill-scope, not pod-scope. | Independent surface. |
| Household admin cap-token | **Out of scope V1.** | Revisit if Inrupt-ACP-mediated alternatives become useful. |

**Rule of thumb:** ACP/WAC becomes the default for "share a resource
with a user." Cap-token remains the explicit choice for power-user
flows, bot/admin delegation, and out-of-band token transport
(WhatsApp, email, etc.).
