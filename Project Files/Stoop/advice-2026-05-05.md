# Stoop — integration advice + high-level design (2026-05-05)

> Synthesis of `02-neighborhood-app`, `apps/neighborhood-v0`, the
> Stoop brainstorm, and the mockup HTML — measured against what the
> SDK + substrates actually ship today. Stays close to existing
> primitives; recommends what to keep / drop / defer; relay section
> is deliberately deeper because the user asked for it.

## TL;DR

- **Stoop ≈ `apps/neighborhood-v0` + a polished UX layer + a small relay-operator kit.** Most of what the brainstorm imagines is already plumbed; the gap is mostly UI + a thin self-hosting story for relays.
- **Relay group-auth is already exactly the model the brainstorm wants:** admin-signed group proofs, public adminPubKey on the relay, no private keys travelling outside the group. Locked Q-E.2 (2026-04-28).
- **Drop the MLS dependency** the brainstorm raises — overkill for a buurtapp, the SDK's `nacl.box` envelope encryption + `GroupProof` + optional `sealedForward` is sufficient for v1.
- **V1 anonymity is social, not cryptographic.** Telegram-style: a user-set `handle` (e.g. `@oosterpoort-bird-23`) is the primary identifier in the UI; real `displayName` is opt-in and revealed per-connection or per-group. WebID stays the stable backstage ID. This is shippable in V1 without new crypto. Cryptographic anonymity (relay can't correlate handle ↔ WebID) stays parked for V2.
- **Privacy / threat model lives in [`privacy-and-safety-2026-05-05.md`](privacy-and-safety-2026-05-05.md)** — closed-beta acceptable items, public-launch blockers, required privacy-notice content.

## The relay, in depth

### What actually ships today

 As of 2026-04-28 the relay has all of these:

1. **Group-gated registration.**
   - On `register`, the client MUST send `groupProof` (a `GroupManager`-issued proof for one of the accepted groups).
   - Relay verifies signature + expiry + (optional) `requiredRole`.
   - If `acceptedGroups` is unset/empty, relay accepts everyone (open mode, backward-compatible).
   - Config shape:
     ```js
     startRelay({
       acceptedGroups: [
         { groupId: 'oosterpoort-skills', adminPubKey: '...', requiredRole: 'member' },
       ],
       // tlsCert, tlsKey for HTTPS/WSS
     });
     ```
   - **The relay never holds a private key for the group.** It only knows the admin's *public* key, used to verify proofs. This is exactly the model the brainstorm asks for.

2. **Group broadcast.**
   - `{ type: 'group-publish', groupId, topic?, envelope }` → relay fans out to all online members of `groupId` (sender excluded).
   - Offline members get queued via the topic-aware buffer.
   - Sender must be a member of `groupId` (verified at register time).

3. **Topic-aware offline queue.**
   - Per-`(addr, topic)` bucket capping. A noisy publisher on `skills:bike-repair` can no longer evict a quiet `skills:tax-help` bucket. Total per-address buffer is `4 × queueCap` as a safety valve.

4. **Multi-recipient (E2b).**
   - `multi-request` → fan-out to N targets, collect responses with timeout. Useful for "ask 5 skill-holders, take fastest". Saves the app from doing this manually.

5. **Push wake (E2c).**
   - `PushTokenRegistry` stores per-address Expo tokens.
   - When a recipient is offline AND has a token, the relay sends a `notification.send` Expo push (throttled to ≤ 1 / 30 s / address).
   - On the device side, `MobilePushBridge` (in `@canopy/react-native`) bridges back into the agent.

6. **TLS / WSS.**
   - Pass `tlsCert` + `tlsKey` and the server listens on HTTPS/WSS.
   - Self-host pattern: terminate TLS at Caddy (auto-Let's-Encrypt) → reverse-proxy to a non-TLS relay on localhost. Brainstorm's recommendation lands here unchanged.

7. **TCP wire shape stays small.** JSON-over-WebSocket. Two extra
   frame types beyond the obvious (`register`, `send`, `message`):
   `group-publish` / `group-publish-ack` / `multi-request` /
   `multi-deliver` / `multi-response`. Easy to reason about.

### Three relay deployment models (the brainstorm got this right)

1. **Default community relay.** A buurtvereniging or a volunteer runs one for the local pool. Lowest friction; SPOF; meets the "it just works" expectation.
2. **Self-host per group.** The brainstorm's launch-site idea. Caddy + relay container + admin key. Right answer for tech-savvy groups who want sovereignty. -> if there is a free service that offers strong enough servers, then that has preference in this phase 
3. **Federated relays (Nostr-style).** Defer. Real value but real complexity.

For v1: ship the **default community relay** + the **self-host kit**.
Federation later if there's demand.

### A concrete onboarding flow that uses what exists today

Reusing `GroupManager.issueInvite` / `redeemInvite`:

1. **Anna creates "Oosterpoort skills":** app calls `GroupManager.create(...)` (or equivalent) → group key-pair on her device. `acceptedGroups` entry submitted to the community relay (just `{groupId, adminPubKey}`).
2. **Anna invites Bob:** `issueInvite(...)` → QR with `{ groupId, adminPubKey, inviteToken, relayUrl }`.
3. **Bob scans:** generates his own device key, calls `redeemInvite(...)`, gets a signed `GroupProof`. Stores it in his local vault.
4. **Bob's agent connects** to the relay with his proof in `register.groupProof`. Verified, accepted.
5. **From this moment** all messages between Anna and Bob (in this group) flow over the relay (or directly via mDNS/BLE if reachable, picked by the SDK's `RoutingStrategy`).

No relay-side key material was created or shared. No MLS handshake.
The brainstorm's intuitions match the existing primitives almost
1-to-1.

## Mapping Stoop brainstorm → keep / drop / defer

### Keep (already in the SDK / substrates)

- ✅ Group capability + signed credential (already implemented).
- ✅ QR invite + redeem (already implemented).
- ✅ mDNS / BLE local discovery — "5 mensen in de buurt" is mostly a UI surface over existing transports.
- ✅ Skill posture (`always` / `negotiable` / `humanInTheLoop`) — already in `skill-match`.
- ✅ Mens en machine-agents naast elkaar — `humanInTheLoop` flag drives the UI label.
- ✅ Push wake for human prompts (notifier + relay push).
- ✅ Profile + skills on the user's pod (item-store + identity-resolver patterns are already there).

### Adapt (small additions, mostly UX)

- 🔧 **Prikbord UI** (not feed). Compose `agent-ui`'s `mountLocalUi` over a static front-end; reuse the H5 `web/` shell. Drop counters / likes / time-pressure cues by design.
- 🔧 **Vraag / Aanbod / Te leen as item-types.** Don't invent a new substrate; extend `item-store`'s item shape with a `kind` enum (`'ask' | 'offer' | 'lend'`). All three live on the same prikbord.
- 🔧 **Handle as primary ID, displayName on reveal.** Telegram-shape (see "Handle / nickname design" below). Profile carries `handle` + optional `displayName`; UI defaults to handle in all groups; `displayName` becomes visible after a `chat-agent` connect-accept handshake or when explicitly opted in for a group.
- 🔧 **"Bekijk als familie / buurt / publiek" preview.** Smart UX move from the brainstorm. Implementation is server-side filtering of the prikbord by the chosen group's `GroupProof`; the UI just sends a different `viewer` parameter.
- 🔧 **Per-group display config.** Brainstorm's "labelen met privelabel waar eigenschappen overgenomen worden" maps onto `identity-resolver`'s member-map: each group can carry its own member-display config (per-group nickname override, per-group reveal state).
- 🔧 **Quotas on the relay** (small server-side addition; per-group rate limits) — needed for self-host safety. Not on identity-resolver, on the relay itself. **V1 deliverable** — see Phase 1.
- 🔧 **Default pod-provider** so a buurtbewoner doesn't have to choose. Same pattern Folio mobile already uses (`DEFAULT_INRUPT_ISSUER`).

### Drop

- ❌ **MLS.** Not needed at this scale; out-of-proportion to the threat model. Use `SecurityLayer`'s envelope encryption.
- ❌ **Five-tier openheidslevels** (publiek / familie / buurt / dorp / specifieke groep) as a hard-coded ladder. The brainstorm itself self-corrects: "capability-based — elke skill heeft een ACL die verwijst naar groepen, en publiek/buurt zijn voorgedefinieerde groepen." Adopt that. Just groups, all the way down.
- ❌ **Star ratings.** Brainstorm flags this; agreed — too transactional for a buurt context.

### Defer (real but later)

- ⏳ **Cryptographic anonymity** (`02-neighborhood-app` parks it; brainstorm assumes it). The genuinely hard piece — relay can't correlate handle ↔ WebID. Plan: V1 ships *social* anonymity only (handle hides name, displayName-on-reveal) + periodic agent-identity rotation (see "Identity rotation" below); V2 unparks Q-H5 for true cryptographic anonymity. **Reword the mockup's "anoniem plaatsen" checkbox.** Honest V1 wording (Dutch): *"Toon alleen mijn handle (server ziet wel je account)"* — or shorter: *"Naam verbergen tot connectie"*. Don't ship the original "Anoniem plaatsen" label in V1; it overpromises.
- ⏳ **Federated relays** (Nostr-style). Defer.
- ⏳ **Relay launch-website / admin GUI**. Defer to a separate "Stoop Relay Kit" deliverable; not blocking the user-facing app.
- ⏳ **Skill chains / ring-trade matchmaking** ("Anna ↔ jij ↔ Bert"). Charming, agent-shaped, defer to V2 once the basics work.
- ⏳ **Skill stories + rich profiles**. UI polish; defer.
- ⏳ **Agent personas / naming**. Defer; align with the AgentHub design when it lands.
- ⏳ **Buurt-resources (boormachine, aanhanger) as first-class non-person agents**. Modelable today as `humanInTheLoop: false` agents owned by a member; defer until v1 ships.

## Handle / nickname design (V1)

> Telegram-shape: a user-defined handle is the primary identifier in
> the UI; the real-name `displayName` is opt-in and only shown after
> a connection or per-group opt-in. WebID stays the stable backstage
> identity. **No new crypto** — this is a UX + data-model change.

### Data model (on the user's pod)

Agent profile gains two fields beside the existing WebID:

```
profile/
  webid              ← stable, never displayed in the UI by default
  handle             ← user-set, lowercase, e.g. 'oosterpoort-bird-23'
                       (uniqueness scoped per-group, not globally — collisions are tolerable)
  displayName        ← optional real name; revealed by handshake or per-group opt-in
  reveals/
    <groupId>        ← per-group reveal state: { showDisplayName: bool, override?: handle }
    <peerWebid>      ← per-peer reveal state (after chat-agent handshake)
```

### Resolution order in the UI

For a given viewer + target:

1. If viewer has a per-peer reveal record for target with `showDisplayName=true` → show `displayName`.
2. Else if target opted in to display-name visibility for this group → show `displayName`.
3. Else show `handle` (and a small "@" prefix to make the pseudonym legible).
4. Never show raw WebID.

### Where this lives in the substrate stack

- `identity-resolver` (L1h) is the natural home — it already exposes a member-WebID-to-display-name map. Extend with:
  - `resolve(viewerWebid, targetWebid, { groupId? })` returning `{ handle, displayName?, isRevealed }`.
  - Per-group nickname override (already partly supported via `MemberMap` config).
- `chat-agent` handshake (L1c) flips a reveal flag in the viewer's per-peer reveal record.
- `item-store` items continue to carry the WebID as author; the UI rendering layer is what applies handle/displayName resolution.

### What doesn't change

- WebID stays the stable network identifier. All signing, capability tokens, group proofs continue to use it.
- `GroupProof`s carry the WebID, not the handle. Handle is purely a UI rendering concern.
- No relay-side changes. No new envelope shape.

This is the cheap version of the brainstorm's "anoniem plaatsen ↔
connectie accepteren" flow. It does not protect against an adversary
correlating handle ↔ WebID via relay traffic — that's V2 territory
(see [`privacy-and-safety-2026-05-05.md`](privacy-and-safety-2026-05-05.md)).

## Substrate fit + rule-of-two

Stoop's needs that aren't yet covered, mapped to whether a substrate
extension or a new substrate is the right answer per the project's
rule-of-two policy.

| Stoop needs… | Where it lives | Rule of two? | Verdict |
|---|---|---|---|
| `kind: 'ask' \| 'offer' \| 'lend'` on an item | extension to `item-store` (L1b) | household app also benefits ('shared shopping list' kind, 'household chores') | **extend item-store** |
| Lend lifecycle: due-date field + return reminder | extension to `item-store` + `notifier` (L1f) | household app likely wants this for shared-resource tracking | **extend item-store + notifier**, no new substrate |
| Handle as primary ID, displayName on reveal | extension to `identity-resolver` (L1h) | every social-shaped app wants this (household, future Folio sharing UX) | **extend identity-resolver** |
| Per-group nickname override | `identity-resolver`'s existing `MemberMap` config | already supported; document the pattern | **no change, document** |
| Per-group rate quotas on the relay | `relay` package (server-side) | every closed-group app deployed to a self-hosted relay needs this | **extend relay**, not a substrate concern |
| "Bekijk als …" group-scoped profile projection | `identity-resolver` + `item-store` query layer | unclear — household *may* want it, not certain yet | **extend in-place for now; extract if a 2nd app emerges** |
| Container + admin-GUI for the relay (Stoop Relay Kit) | new sub-project, NOT a substrate | by definition — operator tool, not an SDK building block | **separate project** |
| Cryptographic anonymity (V2) | new core primitive (anonymous credentials) + maybe a substrate to compose them | only Stoop V2 needs it today | **defer; design when Q-H5 unparks** |
| Multi-relay / Nostr-style federation (V2/V3) | new core mechanism + client-side dedupe | speculative; no second app needs it yet | **defer; rule-of-two not satisfied** |

**Net conclusion:** Stoop V1 triggers two substrate extensions
(`item-store` for `kind` + lend lifecycle, `identity-resolver` for
handle + displayName-on-reveal). Both have second consumers (the
household app), so rule-of-two is satisfied. Plus one server-side
addition (relay quotas), which sits in `@canopy/relay` and is not a
substrate concern. **No new substrate package** is required for V1.

## Moderation v0 (V1 deliverable)

> Closed groups have admins, but until V1 ships moderation skills,
> the only recourse for conflict is "uninstall the app". That's
> not viable. V1 needs a thin, honest moderation layer.

### Skills to ship in V1

| Skill | Caller | Effect |
|---|---|---|
| `removeMember(memberWebid, reason?)` | admin / coordinator | Refuses to issue any further `GroupProof` for this WebID; broadcasts a member-removed event so existing peers stop accepting their messages; relay-side revocation list (admin-signed) blocks fresh registrations. |
| `leaveGroup()` | self | Local: revokes own proof, drops the group from the switcher, deletes group-scoped state from pod (with a "keep my posts visible? Y/N" prompt). |
| `reportPost(itemId, reason)` | any member | Adds a signed report to a per-group `reports/` collection on the admin's pod; admin sees the queue in their UI. |
| `mutePeer(peerWebid)` | any member, local-only | Filters that peer's posts + claims out of the local prikbord without affecting anyone else's view. Pure UI filter; no network effect. |
| `setMemberRole(memberWebid, newRole)` | admin | Re-issues a proof with the new role (e.g. `coordinator`). Existing core `Roles` rank table already supports this. |

### "Remove" semantics — the practical trick

True hard revocation is V2 (deferred per the relay table in this
doc). For V1, the practical "remove" mechanism is:

- **Short-TTL `GroupProof`s** — issue with a 7-day expiry by default. A removed member stays connected for at most 7 days; admin's refusal to re-issue is the actual ban.
- **Relay-side revocation list** — a small admin-signed list of revoked-WebIDs, pulled or pushed periodically; relay rejects fresh `register` from listed WebIDs even if their proof is still valid.

Document this clearly: a removed member can still see new posts for
up to 7 days. For a buurtapp this is acceptable; flag it in the
admin UI when a removal is initiated.

### Where the moderation flows live

- New skills in `apps/stoop/src/skills/moderation.js`.
- Per-group `reports/` collection in pod via `item-store` (with `kind: 'report'`).
- Admin moderation queue rendered in the existing `agent-ui` shell.
- Relay-side revocation list: small extension to `GroupAuthVerifier` (load + check). Same shape as the V1 quotas addition.

### Open moderation questions

- Should reports be visible to all admins of a group, or only the receiving admin?
- Should a member be able to *appeal* a removal? V2 work.
- Multi-admin coordination (two admins disagree) — out of scope for V1.

## Identity rotation (V1, partial)

> Good news: `Agent.rotateIdentity()` is a real, shipped primitive
> in `@canopy/core` (Group FF). Worth using in V1 to take a bite
> out of the "stable WebID enables long-term de-anonymisation" risk
> from the privacy doc, even though full unlinkability stays V2.

### What rotates and what doesn't

| Stays stable | Rotates |
|---|---|
| **Pod WebID** — must, because it anchors the data | **Agent network identity** — Ed25519 keypair the relay sees, peers correlate against |
| Pod-stored profile, items, group registrations | The pubKey on a `GroupProof`, the address the relay routes to |
| User-set `handle` and `displayName` | Group `GroupProof` (re-issued against the new pubKey on rotation) |

So an external observer scraping relay traffic sees the same pubKey
sending in `oosterpoort-skills` for ~30 days, then a different
pubKey, with no on-the-wire correlation between the two (rotation
broadcast is intra-group only). Pod-side correlation is still
possible if the observer also has access to the pod (they shouldn't),
so this is a partial improvement, not a full anonymity story.

### V1 rotation policy

- **Cadence:** every 30 days by default. Configurable per group.
- **Grace period:** 7 days (the SDK default). Old key still valid for a week so unsynced peers can catch up via inline rotation proofs.
- **Trigger:** `Agent.rotateIdentity({ gracePeriodSeconds: 604_800, broadcast: true })` on a scheduled job.
- **Group re-issuance:** on rotation, the agent calls the admin's `requestProofRefresh` skill (new — small addition); admin auto-re-issues if the requester's previous proof was still valid. No human-in-the-loop unless a member is being removed.

### Relay change required

The current `GroupAuthVerifier` accepts a single `groupProof` per
register; it doesn't yet have a "rotation chain" concept. **V1 small
addition:** during the 7-day grace period after rotation, a register
may carry an old `GroupProof` plus an inline `rotation-proof`
linking the new pubKey to the old one (the SDK already produces
these). Relay verifies the chain and accepts. After grace, only the
fresh `GroupProof` is accepted.

### What rotation does NOT solve

- An observer who logs both pre- and post-rotation traffic can
  potentially correlate via timing / volume / reply graphs. True
  unlinkability needs cover traffic + mixnets — V2.
- Pod-side correlation: any party with read access to the user's
  pod sees a stable WebID. Pod ACPs are the defence; rotation is
  not.

## Engineering practice — feedback loop on UX-load-bearing UX

> Some V1 features have parameters that nobody can pre-tune from a
> spec. Shipping a "best guess" and iterating on real usage is the
> only way to reach the right setting.

The single largest example: **push notification cadence and
fan-out**.

- Push too aggressively → spam, churn, "Stoop notifies me 10x/day, deleted".
- Push too cautiously → matches die unanswered, "I posted a vraag and nothing happened".
- The right setting depends on group size, member age distribution, time of day, message kind (vraag vs. aanbod vs. te leen), and individual preference.

**Plan: don't pretend to know the answer.** Ship V1 with sensible
defaults, instrument the metrics, give users dials, and adjust based
on a feedback loop:

1. Default: **push for `humanInTheLoop: true` matches only**, throttled to ≤ 3 per day per member, batched into a digest if more.
2. Per-user opt-out: "no push, I'll check in".
3. Per-group admin dial: "more eager push" / "calmer push" — same skill-match posture, applied at group level.
4. Metrics (locally aggregated, opt-in to share with admin): notifications received, notifications dismissed, vragen answered after push vs. without.
5. **Adjust defaults at V1.5 / V2** based on what real groups do.

Same pattern applies to: lend-return reminder cadence, "who's nearby"
proximity ping rate, group-membership re-key TTL, prikbord fetch
cadence. Anywhere a parameter is in the UX-load-bearing critical
path, plan for the loop.

## "Bekijk als …" preview — design note

The mockup's profile-screen toggle (*"bekijk mij als familie / als
buurt / als publiek"*) is one of the strongest UX ideas in the
brainstorm — privacy settings become concrete. **Defer the correct
implementation to V2.**

Rationale: doing it correctly requires evaluating the user's pod
ACPs from the viewer's perspective. Two approaches:

- **Server-side simulation:** in-app code that mirrors the ACP rules and renders accordingly. Cheap, but drifts from the real pod ACP — what the simulator shows can diverge from what a real viewer sees. Bad outcome: user thinks they hid something; real viewers see it.
- **Real auth from a stand-in identity:** the app authenticates as a member of the previewed group and queries the pod. Correct but slow (extra OIDC round-trip) and requires permission infrastructure that doesn't exist yet.

V1 ships without the toggle. V2 implements it alongside the ACP-correctness
automated tests called out in the privacy doc.

## Recommended high-level design

### Agent topology

- **One agent per user, per group.** The agent's identity is anchored on the user's pod (the "agent profile" idea from the brainstorm); its `GroupProof` for *this* group lives in its vault. This is exactly what neighborhood-v0 does today.
- **Group switcher** lets a user belong to N groups; UI swaps which agent is "active". `apps/neighborhood-v0` already supports this via `--groups <gid1>,<gid2>`.
- **Agents = SDK Agent class.** Handles transport selection (mDNS / BLE / relay / NKN / hop) automatically via `RoutingStrategy`. App code never picks the path.
- **Aligns with Agent Hub direction** (see `Project Files/AgentHub/agent-hub-design-2026-05-05.md`): per-app-agent today, hub-attachable later. Spawn under user root identity via capability tokens — the existing pattern.

### Substrate composition (for reference)

```
apps/stoop  (UI + onboarding + Stoop-specific UX)
   │
   ├── @canopy/skill-match    (vraag/aanbod broadcasting)
   ├── @canopy/item-store     (Vraag/Aanbod/Te leen as items, pod-backed)
   ├── @canopy/identity-resolver  (member display + per-group label config)
   ├── @canopy/agent-ui       (REST + SSE + mountLocalUi)
   ├── @canopy/notifier       (push wake when human decision needed)
   └── @canopy/chat-agent     (chat → connect → reveal flow)
        │
        └── @canopy/core + @canopy/relay + @canopy/react-native (SDK)
```

Direct SDK use should be limited to constructing the `core.Agent`
(per the existing `## Direct SDK use` justification pattern in the
neighborhood-v0 README) and to relay/transport wiring.

## User stories (V1, non-anonymous)

Each story is keyed to the SDK primitive that satisfies it.

1. **As Anna I open Stoop for the first time and have a working profile in 3 minutes.**
   - Default Solid pod issuer pre-selected (hidden behind "advanced"); pod created via Inrupt's auto-provisioning. SDK: `core.Bootstrap`.
2. **As Anna I scan a QR code from a buurtgenoot to join "Oosterpoort skills".**
   - SDK: `GroupManager.redeemInvite` (already shipped; testbed proves it).
3. **As Anna I post "Iemand handig met fietsen?" visible only to Oosterpoort.**
   - Substrate: `skill-match.postRequest` with `group: 'oosterpoort-skills'`. Item written to pod via `item-store` with `kind: 'ask'`.
4. **As Bob I get a push notification on my phone that someone needs a fiets-fixer.**
   - Substrate: `notifier.PushChannel` over `relay.ExpoPushSender`; tap opens the matches screen.
5. **As Bob I can choose to respond, pass, or stay silent.**
   - Substrate: `skill-match` posture: `humanInTheLoop: true` skills present a prompt; `'negotiable'` posture means the user decides.
6. **As Anna I see Bob's response next to "Buurtwerkplaats" (a non-human agent that also matched).**
   - Mens en machine in dezelfde lijst, distinguished by `humanInTheLoop` flag in the UI.
7. **As Anna I chat anonymously with Bob, then accept the connection and see his real profile.**
   - V1: chat is non-anonymous already (just hides full profile until accept). V2: `sealedForward` for true content-blind chat.
8. **As Anna I switch to my "Klusclub" group and see a different prikbord.**
   - Group switcher (already in neighborhood-v0).
9. **As Carol I borrow Jasper's aanhanger for the weekend.**
   - Same flow as a skill request, with `kind: 'lend'`. No new substrate.
10. **As a buurtvereniging I run our own relay.**
    - Future deliverable: Stoop Relay Kit. v1 = community relay + manual self-host via the existing `relay` package + Caddy.
11. **As Anna I'm on a buurt-BBQ and the app shows me 5 people nearby with overlapping skills, no signal needed.**
    - Local-magic: SDK's `MdnsTransport` + `BleTransport` already do peer discovery. UI surfaces it.

## Phased plan

### Phase 1 — Stoop V1 (UX layer over neighborhood-v0)

- Adopt neighborhood-v0 as the engine; rename / fork as needed.
- Add `kind: 'ask' | 'offer' | 'lend'` to item-store items (small additive change).
- Add lend lifecycle: `due` field on items + return-reminder via `notifier`.
- Build the prikbord UI (replace neighborhood-v0's stark `index.html` with the mockup-derived design; same backend skills).
- Default pod issuer + onboarding without "kies een Solid pod".
- **Handle / displayName-on-reveal in `identity-resolver`** (see "Handle / nickname design" above). Includes per-group reveal state + `chat-agent` handshake flips it.
- Group switcher (already there). **Defer "bekijk als …" preview to V2** (see "Bekijk als preview — design note" below).
- Mens/machine label in UI based on `humanInTheLoop`.
- **Moderation v0** — `removeMember`, `leaveGroup`, `reportPost`, `mutePeer`, `setMemberRole` skills + relay-side revocation list + 7-day default proof TTLs. See "Moderation v0" section below.
- **Identity rotation** — every 30 days by default, via `Agent.rotateIdentity()` (already in core); admin auto-re-issues group proofs on rotation. See "Identity rotation" section below. Requires a small `GroupAuthVerifier` change to accept inline rotation proofs during the grace period.
- **Push UX feedback loop** — instrument metrics from day 1; default to conservative push (≤ 3 / day, `humanInTheLoop` only); add per-user / per-group dials. See "Engineering practice — feedback loop" section below.
- **Add per-group rate quotas on the relay** (small addition to `GroupAuthVerifier`). Closed-beta-acceptable defaults; operator-overridable. Required before any external user joins (per [`privacy-and-safety-2026-05-05.md`](privacy-and-safety-2026-05-05.md)).
- Ship the closed-beta privacy notice (content per `privacy-and-safety-2026-05-05.md` § "Closed-beta privacy notice — required content").
- Document the community-relay deployment recipe (Caddy + relay package + admin's `acceptedGroups` config).
- Localisation: shipping with at least `nl` and `en` locale files per the project localisation convention (see `Project Files/conventions/localisation.md`).

### Phase 2 — Stoop Relay Kit

- Containerise the relay (Dockerfile + compose).
- Small admin web UI authenticated with the operator's admin key.
- Admin GUI lets the operator add/remove `acceptedGroups`, see quotas, see connected counts.
- "Launch site" sugar: deploy-to-Hetzner / deploy-to-Fly button equivalents — secondary, charming.

### Phase 3 — V2 features

> All V2 items are designed against the threat model in
> [`privacy-and-safety-2026-05-05.md`](privacy-and-safety-2026-05-05.md).
> Specifically: cryptographic anonymity replaces V1's social
> anonymity; ACP-correctness automated tests; admin re-key +
> revocation flow; push-token rotation.

- **Unpark cryptographic anonymity (Q-H5):** anonymous credentials, identity-blind discovery, single-use proofs so the relay cannot correlate handle ↔ WebID. Replaces V1's social-anonymity-only model.
- **ACP-correctness automated tests** so a misconfigured pod ACL doesn't silently expose private data (per privacy-doc § "Pod ACP misconfiguration").
- **Admin re-key + revocation flow** (per privacy-doc § "Group admin private key").
- **Push-token rotation** (per privacy-doc § "Push tokens identify devices").
- Multi-relay / federation if there's user demand.
- Skill chains / ring-trade matchmaking.
- Richer profiles (skill stories, photos, agent personas).
- Buurt-resources as first-class non-person agents.

## Open questions

1. **Naming.** Stoop = a renamed neighborhood-v0? A sibling app? A new top-level app composing the same substrates? Recommend: new top-level `apps/stoop`, depending on neighborhood-v0's substrate stack but with its own UI and onboarding.
2. **Default pod provider.** Which one(s) ship pre-selected? Folio mobile uses `login.inrupt.com`; Stoop should match unless there's a strong reason to differ.
3. **Community relay operations.** Who runs the v1 default? Do we ship a hosted one for the PoC, or always require self-host?
4. **Quota policy defaults.** What's "X msgs/day, Y MB, Z connections" for a small group on a community relay? Needs measurement once the prikbord generates real traffic.
5. **"Bekijk als …" implementation.** Server-side filter (cheap, but trusts the agent) or pod-ACP-derived view (slow, correct)? V1 = server-side, document the limitation.
6. **Lend / borrow lifecycle.** Skills are timeless; lends have a return date. Item-store needs a soft "due" field, and a nudge from `notifier`. Small substrate adjustment.
7. **Onboarding magic vs Solid honesty.** How loudly do we tell users their data is on a pod? Brainstorm's "decentralisatie als gevoel, niet als architectuur" is sound; needs explicit copy guidelines.
