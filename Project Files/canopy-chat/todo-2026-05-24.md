### ε wave — catch-up by group strategy (NEXT, after δ.2)

**The gap.** Today's catch-up paths are uncontrolled: rehydrator pulls from the local stoop store on every boot, and the reconnect catch-up path silently asks ALL peers for "everything since last seen" — no size signal, no provider notification, no receiver choice. After a week of activity on a high-traffic kring this could move significant bytes blindly. Worse, adding new sync paths (e.g. NKN broadcast, pod range-query) without a single normalization gate risks competing-sync-paths bugs where messages arrive twice or zero times depending on race timing.

**Per-kring strategy.** The kring's `pod` axis already encodes the right policy:
- `pod: 'shared'` → range-query the central pod (the pod is authority; no peer consent needed for reads)
- `pod: 'personal'` / `'none'` → peer-to-peer; this is where the negotiation+notification protocol applies
- `pod: 'hybrid'` → pod primary + peers as backfill

**Slices:**
- **ε.1 — single normalization gate (foundation, M).** Audit every code path that inserts a chat-message into the eventLog (rehydrator, kringChatReceiver, catch-up, future pod range-query) and force them all through one msgId-dedup point. Without this, ε.3+ε.4 race each other. Likely lifts the existing LruSet in kringChatReceiver up to a shared `chatMessageInbox.js` substrate with explicit dedup + ordering guarantees.
- **ε.2 — per-group strategy router (M).** Per-kring catch-up dispatcher reads `circle.policy.pod` and picks the strategy (pod-range-query for shared; peer-negotiation for personal/none; both for hybrid). Wires into the existing reconnect hook.
- **ε.3 — pod range-query catch-up (S/M).** New stoop skill `getMessagesSince(groupId, sinceTs, max?)` over the pod; receiver calls it on reconnect for shared-pod kringen.
- **ε.4 — peer-negotiated catch-up protocol (M).** New peer-message subtypes `catch-up-request{groupId, sinceTs}` + `catch-up-offer{count, sizeBytes, lastTs}` + `catch-up-stream{messages, finished}`. Receiver sends request → providers respond with offers → receiver picks one + sets ceiling (`max N` or `lastDays`) → provider streams.
- **ε.5 — provider notification + receiver choice UI (M).** Provider gets a small notification ("Alice asks for X messages from {kring} — ~Y KB. [Send all] [Send last 50] [Decline]"). Receiver sees a choice when multiple offers arrive ("From: Alice, Bob; ~145 messages; [Get all] [Last 50] [Last 7 days] [Cancel]").

**Sequencing:** ε.1 + ε.2 are foundation — parallelizable. ε.3 unblocks shared-pod kringen quickly (small). ε.4 + ε.5 are the bigger protocol work; do them together since the notification UX and the message shapes are interdependent. Total: ~5 slices, two parallel rounds.
