### Оµ вЂ” manual / real-device tests (TODO once peers exist)

The negotiated flow can't be exercised end-to-end with the current setup (single device, `[catch-up] skipped (0 peers)`). Test plan when a second device joins a kring:

1. **2-device peer kring (`pod: 'personal'` or `'none'`)**:
   - Device A in the same kring as Device B.
   - Close Device A's app for >5 minutes while Device B sends в‰Ө3 chat messages to the kring.
   - Reopen Device A. Expect to see the **"Catching upвЂ¦ X/Y"** indicator briefly in the chat-shell, then the new messages arrive in the bubble stream.
   - Verify in logcat: `catch-up-request` sent, `catch-up-offer` received with non-zero count+sizeBytes, `catch-up-chunk` stream, `catch-up-end` with totalSent matching.
   - Then RE-open without B sending anything new в†’ indicator should be brief + clear with "no-offers" or zero-count (no work to do).

2. **3-device negotiated kring** (when feasible):
   - Devices A + B + C all in the kring.
   - Close A while B + C send. Reopen A.
   - Expect TWO offers (one from B, one from C). Receiver auto-accepts the FIRST one (V1 policy). Verify the second offer is silently ignored (logs only).
   - Future Оµ.6: multi-offer chooser UI; defer until 3-device test exists.

3. **Shared-pod kring (`pod: 'shared'`)** when pod-attached kringen exist:
   - Same close в†’ wait в†’ reopen pattern.
   - Expect the strategy router to log `strategy: 'pod'` and NO peer envelopes; messages arrive via `getMessagesSince` instead.
   - Verify the inbox source-tag is `'pod'` (vs `'catchUp'` for the peer path).

4. **Provider notification (auto-approve off)**:
   - On Device B, set `circle.policy.catchUpAutoApprove: false` for a kring (no UI for this toggle yet вЂ” set via dev-tools or a debug slash).
   - From Device A, trigger a catch-up while B is online.
   - Expect Device B to show a banner: "{Alice} wants to catch up on {kring} вЂ” N messages В· ~M KB" with [Send all] / [Last 50] / [Last 7 days] / [Decline].
   - Verify each button works (declining в†’ A's request times out cleanly; mode-pick в†’ only matching items arrive).

5. **Decline / timeout edges**:
   - Receiver fires request with zero peers online в†’ indicator briefly shows + clears with "no offers".
   - Provider that auto-approves but has zero items for the kring в†’ silent (no offer reply, receiver times out at 3s).
   - Provider that streams then disconnects mid-flight в†’ chunk timeout (10s default) fires, receiver cleans up.

Logs to watch:
- `[chatMessageInbox]` insertion + dedup counters (web + mobile)
- `[catch-up]` strategy + results per kring (per-kring loop log line in the new wiring)
- `[catchUpReceiver]` state machine transitions: `PENDING_OFFERS в†’ ACCEPTED в†’ DONE` / `TIMED_OUT`
- `[catchUpProvider]` request validation + offer compute + stream chunks
