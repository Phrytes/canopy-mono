### Flagged implementation calls from the E-sweep (2026-06-02)

Judgment calls made while shipping E2/E4/E5 ŌĆö recorded so they're auditable and
can be revisited:

- **E4 ŌĆö `chat-nav` repurposed, not adopted as-designed.** The package was built
  for a *separate-page* `returnTo` model the app never used (it uses in-SPA panels
  with a bare `[├Ś]`). Rather than add a redundant button, the floating "ŌåÉ back to
  chat" now **refocuses the originating thread** ŌĆö real value over `[├Ś]`, because
  the logs panel's *[Open in chat]* can switch the active thread. **Mobile has no
  chat-nav consumer by design:** chat-nav is DOM-only, and the RN side panels are
  bottom-sheet Modals (onRequestClose + backdrop + Done) with no thread-switch to
  restore. If a future surface opens a *real separate page*, revisit the original
  URL-`returnTo` path.
- **E5 ŌĆö mobile expand is parity, not a gap-fix.** The "cramped mini-page with only
  `[├Ś]`" gap is web-specific; the RN record bubble already shows every field inline
  at full width. The `RecordDetailModal` is therefore an affordance/parity add for a
  single-column device, not closing a real defect. Full RN render is covered by the
  Detox smoke (no RN unit-render infra in the mobile package); the shared contract
  is unit-tested on web + via the pure `recordExpand.js` predicate.
- **E2 ŌĆö mobile cross-thread reach is shallower than web, by architecture.** Web's
  `runBulkOp` item-changed events ride the EventRouter to *other* threads (OQ-4).
  Mobile has **no filter-based EventRouter**, so mobile bulk = the action fan-out +
  summary only; open panels in other threads are not auto-refreshed from a bulk.
  Acceptable parity-of-capability; full parity waits on a mobile event-router (same
  limitation noted for E3 mobile).
